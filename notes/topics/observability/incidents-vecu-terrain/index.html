    Investigation d'Incidents · 4 Patterns Récurrents

    Quatre patterns reviennent régulièrement en exploitation d'applications Java en production : infrastructure, applicatif, dépendance externe et charge temps réel. Cette page les déroule comme des investigations concrètes, avec les commandes réellement tapées, les sorties typiques et les extraits de logs à reconnaître. Elle se termine par une checklist d'investigation universelle réutilisable comme runbook.

    L'objectif : reconnaître le pattern vite, pour engager la bonne approche au lieu de tout passer en revue sans hypothèse.

    Pattern 1 · Saturation Infrastructure

    Contexte PME · serveur Linux unique · plusieurs applications Tomcat · outillage limité

    Le signal

    Remontées utilisateurs : plusieurs applications hébergées sur le même serveur deviennent lentes puis inaccessibles par intermittence. Pas d'alerte automatique (pas de Prometheus à l'époque), c'est le support qui escalade.

    Le déroulé

    Première chose en se connectant en SSH : prendre la température de la machine.

    $ uptime
 14:22:47 up 89 days,  3:11,  2 users,  load average: 14.83, 9.21, 5.40

    Load à 14 sur une machine 4 cœurs : saturation nette, et le 14.83 (1 min) bien au-dessus du 5.40 (15 min) → l'incident est récent et monte.

    $ free -m
              total        used        free      shared  buff/cache   available
Mem:           7820        7102         210          12         507         198
Swap:          2047        1980          67

    available à 198 Mo et swap quasi plein (1980/2047) : la RAM est saturée, la machine swappe à mort, d'où les lenteurs.

    $ df -h
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        20G   19G  512M  98% /
/dev/sda2        50G   12G   36G  25% /opt

    / à 98% : la partition système est quasi pleine. Souvent le coupable, ce sont les logs.

    $ du -sh /var/log/* | sort -h | tail -5
1.2G    /var/log/messages
2.1G    /var/log/audit
8.7G    /var/log/tomcat

    catalina.out de Tomcat à plusieurs Go : il n'a jamais été tourné (pas de logrotate). Il remplit le disque ET la lecture des derniers Go est lente.

    $ ps aux --sort=-%mem | head -4
USER  PID  %CPU %MEM    VSZ    RSS  STAT  COMMAND
tomcat 4821 88.0 71.2 938124 5701632 Sl   /usr/bin/java ...catalina...
tomcat 4990  4.1 12.0 3120044  960012 Sl   /usr/bin/java ...

    Un process Java à 71% de RAM et 88% CPU. Il tire toute la machine.

    Ce qu'on trouve dans les logs

    $ tail -200 /var/log/tomcat/catalina.out | grep -iE "outofmemory|too many|socket"

    Et là, les classiques :

    SEVERE [http-nio-8080-Acceptor] org.apache.tomcat.util.net.Acceptor.run
  Socket accept failed
  java.net.SocketException: Too many open files
    at java.base/sun.nio.ch.Net.accept(Native Method)
    ...

Exception in thread "http-nio-8080-exec-214"
  java.lang.OutOfMemoryError: unable to create new native thread

    Mots-clés à reconnaître :

    Message Ce que ça signifie
    Too many open files File descriptors épuisés (ulimit -n trop bas vs charge)
    unable to create new native thread Plus de mémoire pour les stacks des threads OU ulimit -u atteint
    Java heap space Heap saturée, fuite ou -Xmx sous-dimensionné
    GC overhead limit exceeded La JVM passe son temps en GC sans rien libérer

    Vérification des limites :

    $ cat /proc/4821/limits | grep -iE "open files|processes"
Max open files            1024    4096    files
Max processes             4096    4096    processes

    Max open files à 1024 : ridiculement bas pour un Tomcat qui sert plusieurs applis. Chaque connexion HTTP, chaque fichier de log, chaque socket DB consomme un descripteur.

    Cause racine probable

    Combinaison classique : logs non rotés qui saturent / + ulimit -n trop bas qui plafonne les connexions, le tout sous une charge qui dépasse ce que la machine peut tenir. La saturation mémoire (swap) finit d'achever les temps de réponse.

    Résolution

    À chaud, contenir vite :

    # Libérer du disque immédiatement (tronquer le gros log sans le supprimer)
$ : > /var/log/tomcat/catalina.out      # vide le fichier, garde l'inode

# Redémarrage propre et ordonné, scripté
$ systemctl stop tomcat
$ sleep 5
$ ps -ef | grep -i tomcat               # vérifier qu'il est bien mort
$ systemctl start tomcat
$ tail -f /var/log/tomcat/catalina.out  # surveiller le redémarrage

    À froid, corriger durablement :

    # Relever ulimit (systemd unit ou limits.conf)
LimitNOFILE=65535
LimitNPROC=8192

# Activer logrotate sur catalina.out
/var/log/tomcat/catalina.out {
    daily
    rotate 14
    compress
    copytruncate
    missingok
}

    Limite assumée

    Sans heap dump capturé au moment du pic ni métriques historiques, la cause racine exacte (fuite applicative ? simple sous-dimensionnement ? pic de trafic ?) reste une hypothèse. La séquence se réduit à "contenir proprement, corriger les évidences (ulimit, logrotate), observer si ça se reproduit".

    Enseignement

    Discipline du redémarrage : jamais relancer dans la panique. Toujours capturer un état avant (ps, free, df, derniers logs), scripter la séquence d'arrêt/redémarrage pour qu'elle soit ordonnée et reproductible. Les évidences infra (ulimit, logrotate) doivent être corrigées même si elles ne sont pas LA cause : elles enlèvent du bruit pour la prochaine fois.

    Pattern 2 · Anomalie Applicative Apparente

    Client exigeant sur son domaine · logs centralisés (Splunk) · cache distribué (Infinispan / JGroups)

    Le signal

    La supervision lève une alerte : volume de logs anormalement élevé sur un composant de cache distribué, avec des messages WARN / ERROR récurrents. Personne ne se plaint côté métier, mais le bruit inquiète et un ticket urgent est ouvert.

    Le déroulé

    Première étape : ne pas se précipiter sur la machine, mais qualifier l'anomalie dans les logs centralisés. Requêtes Splunk (SPL) :

    index=prod sourcetype=tomcat host=app-* "Infinispan" (WARN OR ERROR)
| timechart span=5m count by host

    → Pic de logs concentré sur une fenêtre de ~20 min, réparti sur tous les nœuds du cluster (pas un seul host fautif). Indice fort : c'est un événement cluster-wide, pas un bug local.

    index=prod sourcetype=tomcat "ISPN" OR "JGRP"
| rex "(?<code>ISPN\d+|JGRP\d+)"
| stats count by code
| sort -count

    → Top des codes : ISPN000093, ISPN000094, ISPN000310, JGRP000032. Ce sont des codes de changement de vue cluster et de rebalance, pas des codes de corruption ou de perte de données.

    Ce qu'on trouve dans les logs

    Extraits typiques côté Infinispan / JGroups :

    INFO  [org.infinispan.remoting.transport.jgroups.JGroupsTransport]
  ISPN000094: Received new cluster view for channel app-cache:
  [app-01|7] (3) [app-01, app-02, app-03]

WARN  [org.jgroups.protocols.pbcast.GMS]
  app-02: failed to collect all ACKs (expected=3) for view [app-01|7]
  after 2000ms, missing ACKs from [app-03]

INFO  [org.infinispan.topology.ClusterTopologyManagerImpl]
  ISPN000310: Starting cluster-wide rebalance for cache sessions,
  topology id 14

WARN  [org.jgroups.protocols.UDP]
  app-03: discarded message from non-member app-99

    Lecture : un nœud a temporairement quitté puis rejoint le cluster (GC pause longue ? micro-coupure réseau ?), ce qui a déclenché un recalcul de vue (new cluster view) et un rebalance des données du cache. Pendant ces quelques secondes, JGroups logge abondamment en WARN. C'est verbeux mais sain.

    En parallèle, inspection JVM sur un nœud pour écarter une vraie pathologie :

    # Le nœud a-t-il subi une longue pause GC qui l'aurait fait sortir du cluster ?
$ jstat -gcutil <PID> 2000 5
  S0     S1     E      O      M     YGC   YGCT    FGC   FGCT     GCT
  0.00  31.20  68.4   54.1  96.2   1240  38.412   12   28.910  67.322

# Thread dump : des threads bloqués ?
$ jstack <PID> | grep -A5 "JGroups\|Infinispan" | head -30

    jstat montre 12 Full GC cumulant 28s : si l'un d'eux a duré >5s, il a pu geler le nœud assez longtemps pour que JGroups le considère temporairement mort → d'où le rebalance. Hypothèse cohérente.

    Cause racine

    Comportement normal du cache distribué lors d'une reconfiguration de cluster, amplifié par une pause GC sur un nœud. Aucune perte de données, aucun impact métier. Le volume de logs WARN est inhérent au protocole JGroups pendant un merge/rebalance.

    Résolution

    Rapport d'analyse à la direction technique :

    • Pas de patch applicatif nécessaire : rien de cassé
    • Recommandation 1 : abaisser le niveau de log JGroups de WARN à ERROR sur les catégories org.jgroups.protocols.* pour réduire le bruit
    • Recommandation 2 : surveiller les pauses GC (-Xlog:gc*) pour éviter que des Full GC longs ne sortent les nœuds du cluster — piste d'amélioration réelle

    Ticket clôturé sans correctif urgent, avec deux recommandations d'hygiène.

    Effet de bord

    Une analyse solide qui distingue le bruit de la vraie panne crée de la confiance. Concrètement : délégation des tickets N2 suivants, pris en charge directement côté Build jusqu'à la livraison des correctifs.

    Enseignement

    Un incident bien diagnostiqué n'a pas besoin de correctif pour être valorisé. Le livrable, c'est le rapport rassurant : "voilà ce que c'est, voilà pourquoi ce n'est pas grave, voilà comment réduire le bruit". Savoir dire "c'est normal" avec preuves à l'appui est aussi précieux que savoir patcher.

    Pattern 3 · Cause Externe Non Communiquée

    Grande structure / secteur public · équipes cloisonnées · dépendance DB

    Le signal

    Incident intermittent en production, difficile à reproduire. Symptômes flous : certaines requêtes échouent ou traînent, sans régularité. Pas de signal franc côté infra ou applicatif.

    Le déroulé

    Investigation méthodique du bas vers le haut, qui ne trouve rien d'anormal :

    # Machine : RAS
$ uptime          # load normal
$ free -m         # mémoire OK
$ df -h           # disque OK

# JVM : RAS au premier abord
$ jstat -gcutil <PID> 5000 5    # GC sain, pas de Full GC en rafale

# Heap dump à un moment de lenteur, analysé dans Eclipse MAT
$ jmap -dump:live,format=b,file=/tmp/heap.hprof <PID>
# → Leak Suspects : rien de probant. Pas de fuite.

    Puis le test qui change tout : reproduire l'incident hors production. Sur l'environnement de qualif, avec le même build, le même jeu de données... impossible de répliquer. Tout fonctionne parfaitement.

    Quand un incident ne se reproduit QUE en prod, ce n'est généralement pas le code. C'est l'environnement : réseau, dépendance externe, configuration spécifique prod.

    Ce qu'on trouve dans les logs (en cherchant au bon endroit)

    En filtrant finement les logs applicatifs sur les moments de lenteur :

    ERROR [http-nio-8080-exec-88] o.s.jdbc.support.SQLErrorCodesFactory
  HikariPool-1 - Connection is not available, request timed out
  after 30000ms

WARN  [HikariPool-1 housekeeper]
  HikariPool-1 - Thread starvation or clock leap detected

java.sql.SQLRecoverableException: IO Error: Connection reset by peer
  at oracle.jdbc.driver.T4CConnection...

ORA-03113: end-of-file on communication channel
ORA-12170: TNS:Connect timeout occurred

    Mots-clés à reconnaître :

    Message Ce que ça révèle
    Connection is not available, request timed out Pool DB épuisé : toutes les connexions occupées ou mortes
    Connection reset by peer La DB (ou un firewall entre les deux) a coupé la connexion
    ORA-03113: end-of-file on communication channel Lien réseau vers Oracle rompu en cours de requête
    ORA-12170: TNS:Connect timeout Impossible d'établir la connexion dans le délai

    Le faisceau pointe vers une instabilité de la connexion à la base, et uniquement en prod.

    La démarche qui débloque : sortir des logs

    Plutôt que clôturer en "non reproductible", élargir l'enquête aux équipes adjacentes :

    • Équipe exploitation : des changements réseau récents ?
    • Équipe DBA : quelque chose bougé côté base ?
    • Équipe hébergement / réseau : firewall, routage ?

    C'est au téléphone, pas dans les logs, que la pièce manquante apparaît : une partie de la base de données avait été migrée en parallèle par une équipe distante, sans que l'information remonte officiellement au moment de l'incident. La connexion instable venait de cette bascule en cours.

    Cause racine

    Migration de base non communiquée par une équipe distante. La connexion DB devenait instable pendant les fenêtres de bascule. Côté technique : la cause a été trouvée avant la communication officielle. Côté process : symptôme d'une lacune de communication inter-équipes typique des grandes structures cloisonnées.

    Effet de bord positif

    Un incident dont la cause était externe et non communiquée a servi de déclencheur pour robustifier le processus de livraison : scripter l'intégralité des étapes de mise en production, là où on s'appuyait jusqu'alors sur des fichiers Excel décrivant les étapes manuelles. Démarche que l'on qualifierait aujourd'hui de pré-DevOps.

    Enseignement

    Sur les incidents qui résistent à l'analyse technique, surtout ceux qui ne se reproduisent qu'en prod, l'enquête se gagne souvent au téléphone. Réflexe à garder : dès qu'on tape un mur côté technique pur, élargir le périmètre aux équipes adjacentes. La cause d'un incident n'est pas toujours dans le système qu'on surveille.

    Pattern 4 · Saturation Sous Charge Temps Réel

    Plateforme transactionnelle critique · SLA serré · astreinte 24/7 · le serveur va bien mais le métier tombe

    Ce pattern est le plus exigeant. Contrairement aux trois précédents, l'urgence est temporelle : chaque minute de latence = des transactions perdues, un impact métier direct. Le piège classique : tous les voyants infra sont verts et pourtant le service est dégradé.

    Le signal

    Alerte automatique : la latence p99 d'un endpoint critique passe de ~150 ms à plusieurs secondes. Le taux de timeout côté appelant grimpe. Sur un flux transactionnel, c'est P1 immédiat, astreinte réveillée.

    Le déroulé · descendre malgré les voyants verts

    # [1] Infra : tout est vert, et c'est ça le piège
$ uptime
 03:14:22 up 210 days, load average: 2.10, 1.95, 1.80   # 8 cœurs → OK
$ free -m
              total   used   available
Mem:           32000  18000      12000                  # RAM OK
# CPU bas, RAM dispo, disque OK → un débutant conclut "le serveur va bien"
# et passe à côté de l'incident

    Le serveur est sain. Mais le service est dégradé. C'est exactement là que la connaissance applicative fait la différence : on ne s'arrête pas aux métriques machine.

    # [2] Applicatif : où sont les threads ?
$ for i in 1 2 3; do jstack <PID> > /tmp/dump-$i.txt; sleep 10; done

# Compter les threads bloqués au même endroit dans les 3 dumps
$ grep -c "BLOCKED" /tmp/dump-*.txt
/tmp/dump-1.txt:182
/tmp/dump-2.txt:185
/tmp/dump-3.txt:184

# Sur quoi bloquent-ils ?
$ grep -A3 "BLOCKED\|WAITING" /tmp/dump-1.txt | grep -i "hikari\|getConnection\|socketRead" | sort | uniq -c | sort -rn
   180 at com.zaxxer.hikari.pool.HikariPool.getConnection
     2 at java.net.SocketInputStream.socketRead0

    180 threads sur 200 bloqués sur HikariPool.getConnection : tous les threads applicatifs attendent une connexion à la base qui ne vient jamais. Le pool est saturé.

    # [3] Pourquoi le pool est-il vide ? Métriques HikariCP (via Micrometer/Actuator)
$ curl -s localhost:8080/actuator/metrics/hikaricp.connections.pending | jq .
{ "name": "hikaricp.connections.pending", "measurements": [{"value": 178}] }
$ curl -s localhost:8080/actuator/metrics/hikaricp.connections.active | jq .
{ "name": "hikaricp.connections.active", "measurements": [{"value": 50}] }
# 50/50 actives, 178 en attente → les connexions sortent mais ne reviennent pas

    -- [4] Côté base : qu'est-ce qui retient les connexions ?
SELECT sql_id, COUNT(*) AS sessions, MAX(seconds_in_wait) AS max_wait
FROM v$session
WHERE status = 'ACTIVE' AND username = 'APP_USER'
GROUP BY sql_id ORDER BY sessions DESC;
--  SQL_ID         SESSIONS   MAX_WAIT
--  4f9j2k8s...      48          37     ← 48 sessions sur la même requête, 37s d'attente

-- Le plan d'exécution de cette requête
EXPLAIN PLAN FOR <la_requete>;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
--  | Operation             | Name            |
--  | TABLE ACCESS FULL     | TRANSACTIONS    |   ← FULL SCAN au lieu d'INDEX RANGE SCAN

    Ce qu'on trouve dans les logs

    WARN  [HikariPool-1 housekeeper]
  HikariPool-1 - Connection is not available, request timed out after 30000ms
  (total=50, active=50, idle=0, waiting=178)

ERROR [http-nio-8080-exec-200] c.app.payment.AuthController
  org.springframework.dao.QueryTimeoutException: JDBC query timed out

    Côté monitoring métier (Grafana), le consumer lag explose ou le taux d'autorisation chute — le signe que le flux n'est plus traité en temps réel.

    Cause racine type

    Une requête devenue lente (plan d'exécution dégradé après une mise à jour de statistiques, un index manquant, une volumétrie qui a franchi un seuil) monopolise les connexions. Comme chaque transaction attend une connexion, toute la plateforme ralentit en cascade alors que l'infra est parfaitement saine.

    Résolution

    À chaud, restaurer le service :

    -- Forcer le bon plan en attendant (hint)
SELECT /*+ INDEX(transactions tx_merchant_date_idx) */ ...

-- OU tuer les sessions qui bloquent si une requête est partie en vrille
ALTER SYSTEM KILL SESSION 'sid,serial#' IMMEDIATE;

    À froid, corriger durablement : reconstruire l'index ou figer le plan (SQL Plan Baseline), comprendre pourquoi les stats ont changé, ajuster le connection-timeout Hikari pour échouer vite plutôt que de tout faire attendre 30s.

    Enseignement

    Sur une plateforme temps réel, le monitoring métier prime sur le monitoring infra. Un serveur vert ne disculpe rien : si les threads applicatifs attendent tous une connexion DB, le service est mort même avec un CPU à 5%. La capacité à descendre du métier (latence p99) jusqu'à la cause racine (plan SQL) en passant par les threads et le pool est exactement ce qui distingue un responsable d'applications d'un opérateur infra. Et sur ces incidents, chaque minute compte : il faut un runbook prêt, pas une investigation improvisée.

    Lecture transverse · choisir le bon pattern

    Pattern Signal Outils centraux Cause type Posture clé
    Saturation infra Lenteurs générales, swap top, free, df, ulimit, catalina.out FD épuisés / disque plein / RAM saturée Redémarrage scripté + corriger les évidences
    Anomalie applicative Logs WARN en volume Splunk, jstack, jstat Comportement cluster normal + GC Rapport "c'est normal", preuves à l'appui
    Cause externe Intermittent, prod-only logs DB, contacts équipes Dépendance migrée non communiquée Investigation hors logs, au téléphone
    Charge temps réel Latence p99, infra verte jstack ×3, HikariCP, v$session Pool DB saturé par requête lente Monitoring métier > infra, runbook prêt

    Avec la maturité, le niveau d'investigation se déplace du bas (machine) vers le haut (process et dépendances), et l'outillage s'étoffe. Le réflexe constant reste : ne pas relancer dans la panique, qualifier avant de corriger, capturer ce qui peut servir au postmortem.

    Le point commun des quatre

    Dans les quatre cas, la première hypothèse évidente n'était pas la cause finale :

    • Pattern 1 : "c'est la RAM" → en partie, mais ulimit et logrotate étaient les vrais leviers durables
    • Pattern 2 : "le cache plante" → non, il se reconfigurait normalement
    • Pattern 3 : "c'est notre code" → non, une migration externe non communiquée
    • Pattern 4 : "le serveur va bien" → justement, et c'est le piège : tout bloquait sur le pool DB

    D'où la règle : qualifier avant de conclure. Les tableaux de bord verts ne disculpent pas, et les logs alarmants ne condamnent pas. Seul le faisceau d'indices tranche.

    Checklist d'investigation universelle

    Un runbook réutilisable, à dérouler dans l'ordre quand une alerte tombe. L'idée n'est pas de tout exécuter à chaque fois, mais d'avoir un fil conducteur pour ne rien oublier sous pression.

    Étape 0 · Triage (30 secondes)

    uptime                      # charge récente vs ancienne (1min vs 15min)
date                        # noter l'heure exacte de l'incident
# Y a-t-il eu un déploiement récent ? (< 30 min = suspect n°1)

    Étape 1 · Infrastructure

    top / htop                  # CPU, processus chauds
free -m                     # RAM : lire "available", swap actif ?
df -h                       # disque : / et /var pleins ?
du -sh /var/log/* | sort -h # logs qui saturent ?
ps aux --sort=-%mem | head  # gros consommateurs
cat /proc/<PID>/limits      # ulimit (open files, processes)
ss -tan | awk '{print $1}' | sort | uniq -c   # états TCP (CLOSE_WAIT ?)

    Étape 2 · JVM

    jps -lv                     # identifier le PID + flags
jstat -gcutil <PID> 2s 5    # GC : Old gen qui ne redescend pas ? Full GC en rafale ?
jstack <PID> ×3 (10s)       # threads : BLOCKED au même endroit ?
jmap -dump:live,...         # si OOM ou fuite suspectée → Eclipse MAT

    Mots-clés à chercher dans un thread dump : getConnection (pool DB), socketRead0 (appel réseau bloquant), synchronized répété (contention), Found one Java-level deadlock.

    Étape 3 · Application / serveur web

    tail -f catalina.out                          # erreurs applicatives
awk '{print $9}' localhost_access_log* | sort | uniq -c | sort -rn   # codes HTTP
# JMX/Actuator : threads Tomcat busy, pool connexions

    Étape 4 · Base de données

    # Métriques pool : hikaricp.connections.pending > 0 = saturation

    SELECT ... FROM v$session WHERE status='ACTIVE';   -- sessions actives
SELECT blocking_session, sid FROM v$session WHERE blocking_session IS NOT NULL;  -- locks
EXPLAIN PLAN FOR <query>;                           -- plan dégradé ?

    Étape 5 · Dépendances externes

    mtr -rwc 50 <dépendance>                # latence/perte réseau
curl -w "DNS:%{time_namelookup} TCP:%{time_connect} SSL:%{time_appconnect} Total:%{time_total}\n" ...
echo | openssl s_client -connect host:443 | openssl x509 -noout -dates   # cert expiré ?

    Étape 6 · Métier (si dashboards dispo)

    # Le flux métier est-il traité ? (taux d'autorisation, latence p99, consumer lag)
histogram_quantile(0.99, rate(endpoint_duration_seconds_bucket[5m]))

    Étape 7 · Hors technique

    Si les 6 étapes ne donnent rien : appeler les équipes adjacentes (exploitation, DBA, réseau, équipes distantes). Vérifier les changements récents non communiqués (migration, bascule, maintenance).

    Règle d'or finale

    Ne jamais redémarrer dans la panique. Capturer un état (thread dump, métriques, logs de la fenêtre) avant toute action corrective. Un redémarrage efface les preuves : on perd le postmortem.
