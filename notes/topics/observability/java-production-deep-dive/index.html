    Java en Production · Deep-Dive

    Cette page concentre les éléments qu'un ingénieur RUN doit connaître pour gérer une application Java en production critique : modèle mémoire, garbage collectors, tuning, analyse d'OOM, threads, agents et outils d'analyse offline.

    Le fil rouge : rester debout sous charge et diagnostiquer vite quand ça casse.

    Modèle mémoire de la JVM

    ┌────────────────────────────────────────────────────────────────┐
│                       JVM PROCESS (RSS)                         │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    HEAP (-Xms / -Xmx)                     │  │
│  │  ┌────────────────────┐  ┌─────────────────────────────┐ │  │
│  │  │   Young Generation  │  │      Old Generation         │ │  │
│  │  │  ┌──────┐ ┌─┐ ┌─┐  │  │                             │ │  │
│  │  │  │ Eden │ │S│ │S│  │  │  objets long-vivants        │ │  │
│  │  │  └──────┘ └─┘ └─┘  │  │                             │ │  │
│  │  └────────────────────┘  └─────────────────────────────┘ │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  ┌─────────────────┐  ┌──────────────────┐  ┌───────────────┐ │
│  │   Metaspace      │  │ Code Cache (JIT) │  │ Thread Stacks │ │
│  │  (classes, etc.) │  │                  │  │   (-Xss)      │ │
│  └─────────────────┘  └──────────────────┘  └───────────────┘ │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ OFF-HEAP : DirectByteBuffer, Native, NIO, libs JNI        │  │
│  └──────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘

    Heap

    Limité par -Xms (taille initiale) et -Xmx (taille maximale). Découpé en 2 grandes générations :

    Zone Rôle Cycle de vie typique
    Eden Tous les nouveaux objets y atterrissent Minor GC fréquent les évacue
    Survivor (S0/S1) Objets ayant survécu à 1+ Minor GC Promotion vers Old après N passages
    Old (Tenured) Objets long-vivants (caches, sessions) Major / Full GC les nettoie

    Principe générationnel : 90% des objets meurent jeunes. Concentrer le GC sur Eden = beaucoup d'objets nettoyés pour peu de travail.

    Metaspace

    Stocke les métadonnées des classes chargées (depuis Java 8, remplace PermGen).

    # Configurer une borne (sinon limité par la RAM machine)
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m

    Une fuite Metaspace typique : ClassLoader non libéré (hot-redéploiement Tomcat, libs qui créent des proxys dynamiques sans bornes). Signal d'alarme : Metaspace qui croît indéfiniment dans jstat -gc.

    Off-heap

    Pas géré par le GC. Sources : - DirectByteBuffer (NIO, Netty) · libéré par finalizer, peut traîner - JNI (libs natives) - Compressed class space - Thread stacks (par défaut 512 Ko à 1 Mo × nb threads)

    Conséquence importante : un processus Java avec -Xmx8g peut consommer 10-12 Go de RSS sur Linux. C'est normal. Ne pas paniquer si top montre plus que le -Xmx.

    Garbage Collectors · choisir le bon

    GC Disponibilité Forces Faiblesses Quand l'utiliser
    Serial Toutes versions Simple, peu de mémoire Stop-the-world long CLI, batch, embarqué
    Parallel (Throughput) Toutes versions Débit maximum Pauses non bornées Batch hors ligne
    CMS (Concurrent Mark Sweep) Déprécié Java 9, supprimé Java 14 Pauses courtes (historique) Fragmentation, fin de vie Migration vers G1 obligatoire
    G1 (Garbage First) Java 8+, défaut Java 9+ Pauses prévisibles, gros heaps Légèrement moins de débit Choix par défaut prod
    ZGC Java 11+ (stable 15+) Pauses < 10 ms, heaps multi-To Plus de RAM utilisée Latence critique, gros heaps
    Shenandoah Java 12+ (Red Hat) Pauses très courtes Moins répandu Alternative ZGC, faible latence
    Epsilon Java 11+ Pas de GC Pour benchmark uniquement Tests de performance pures

    Choix typiques en paiement / temps réel

    # JVM 11 LTS : G1GC par défaut, suffisant pour 95% des cas
-XX:+UseG1GC

# JVM 17/21 LTS sur heap > 16 Go avec exigence latence : ZGC
-XX:+UseZGC -XX:+UseLargePages

# Forcer le débit (batch) : Parallel
-XX:+UseParallelGC

    Tuning GC concret · flags essentiels

    Dimensionner le heap

    # Toujours Xms = Xmx en production pour éviter les retailles
-Xms8g -Xmx8g

# Mémoire en %, plus moderne (JVM 10+, utile en conteneur)
-XX:InitialRAMPercentage=70 -XX:MaxRAMPercentage=70

# Important en conteneur : la JVM doit voir les limites cgroup
-XX:+UseContainerSupport   # actif par défaut JVM 10+

    G1GC tuning

    -XX:+UseG1GC

# Objectif de pause (la JVM ajuste les régions pour atteindre la cible)
-XX:MaxGCPauseMillis=200

# Pourcentage d'occupation old gen qui déclenche un Mixed GC concurrent
-XX:InitiatingHeapOccupancyPercent=45

# Taille des régions G1 (1 Mo à 32 Mo, puissances de 2)
-XX:G1HeapRegionSize=16m

# Démarrer concurrent marking plus tôt pour éviter Full GC
-XX:G1ReservePercent=15

    ZGC tuning

    -XX:+UseZGC

# Bornes mémoire identiques à Xmx
-Xmx16g -Xms16g

# Logs GC ZGC plus verbeux
-Xlog:gc*,gc+heap=debug:file=/var/log/app/gc.log

    Logs GC · configuration et lecture

    Configuration moderne (JVM 9+)

    # Format unifié `-Xlog:`
-Xlog:gc*:file=/var/log/app/gc.log:time,uptime,level,tags:filecount=10,filesize=100M

# Décomposition :
# gc*                 → tous les tags GC
# file=...            → vers fichier
# time,uptime,level,tags → format de chaque ligne
# filecount=10        → rotation sur 10 fichiers
# filesize=100M       → taille par fichier

    À surveiller

    Métrique Seuil d'alarme Action
    Durée Full GC > 1 seconde Tuner ou changer de GC
    Fréquence Full GC > 1 par minute Pression mémoire, fuite probable
    Heap après Full GC > 70% sustained Fuite confirmée, dumper le heap
    Promotion failure tout occurrence Mixed GC trop tardif, ajuster IHOP
    Concurrent Mark Sweep abort sustained Old gen trop pleine au démarrage du cycle

    Outils d'analyse offline

    • GCViewer · desktop, gratuit, lecture humaine des logs unifiés
    • GCEasy (gceasy.io) · upload du log, analyse automatique avec recommandations
    • Garbage Cat · CLI, parsing rapide
    • JIT Watch · pour le code généré par le JIT

    OutOfMemoryError · les 8 types et leurs causes

    Tous les OOM ne se valent pas. La JVM précise toujours le type dans le message · c'est la clé pour diagnostiquer.

    Type Message JVM Cause typique Solution
    Java heap space OutOfMemoryError: Java heap space Fuite ou heap sous-dimensionnée Heap dump + MAT, augmenter -Xmx
    GC overhead limit OutOfMemoryError: GC overhead limit exceeded > 98% du temps en GC pour < 2% libéré Même que ci-dessus, GC déjà à bout
    Metaspace OutOfMemoryError: Metaspace Fuite ClassLoader (hot-deploy Tomcat, proxys dynamiques) Augmenter MaxMetaspaceSize, traquer le leak
    Direct buffer memory OutOfMemoryError: Direct buffer memory Trop de DirectByteBuffer (Netty, NIO) Augmenter -XX:MaxDirectMemorySize, fix code
    Unable to create native thread OutOfMemoryError: unable to create new native thread Trop de threads vs limite OS Réduire pools, augmenter ulimit -u
    Requested array size exceeds VM limit OutOfMemoryError: Requested array size exceeds VM limit Allocation tableau > 2^31 éléments Bug applicatif, lecture massive non streamée
    Compressed class space OutOfMemoryError: Compressed class space Trop de classes chargées Augmenter CompressedClassSpaceSize
    Reason stack_trace_with_native_method OutOfMemoryError: reason ... Lib native qui crashe Vérifier libs JNI, agents natifs

    Auto-dump sur OOM (à mettre en prod)

    -XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/dumps/heap-%p-%t.hprof
-XX:OnOutOfMemoryError="/scripts/notify-oom.sh"
-XX:+ExitOnOutOfMemoryError    # crash propre plutôt que zombi

    Thread tuning

    Taille de la stack

    # Par défaut : 1 Mo. Réduire à 512k pour héberger plus de threads
-Xss512k

# Calcul : 1000 threads × 1 Mo = 1 Go off-heap juste pour les stacks

    Pool Tomcat

    <!-- server.xml -->
<Connector port="8080"
           maxThreads="400"          <!-- threads disponibles -->
           minSpareThreads="50"
           acceptCount="100"         <!-- file d'attente avant rejet -->
           connectionTimeout="20000"
           keepAliveTimeout="60000"/>

    Règle empirique : - maxThreads ≈ 200 à 800 selon mémoire dispo et type de workload - Workload bloquant (DB lente, banque acquéreur) → augmenter - Workload non bloquant → réduire et utiliser de l'async (WebFlux, virtual threads)

    Pool HikariCP (DB)

    spring.datasource.hikari:
  maximum-pool-size: 50
  minimum-idle: 10
  connection-timeout: 3000
  validation-timeout: 2000
  max-lifetime: 1800000
  leak-detection-threshold: 60000   # warn si connexion sortie > 60s

    Règle empirique : pool-size = (nb cores × 2) + nb disques est le point de départ classique. Trop grand = pas mieux, contrairement à l'intuition (contention sur la DB).

    Virtual Threads (Java 21+)

    Game changer pour les workloads bloquants.

    // Avant (Java 17)
ExecutorService pool = Executors.newFixedThreadPool(200);

// Avec virtual threads (Java 21)
ExecutorService pool = Executors.newVirtualThreadPerTaskExecutor();
// Chaque tâche = 1 virtual thread, peut en avoir des millions

    Bénéfice direct : un appel bloquant vers une banque acquéreur ne monopolise plus un thread carrier. Pas besoin de WebFlux pour scaler.

    Compatibilité Tomcat 11+ : flag useVirtualThreadsForServletInvocations.

    JVM flags essentiels en production

    # === Memory & GC ===
-Xms8g -Xmx8g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/dumps/
-XX:+ExitOnOutOfMemoryError

# === Logs GC ===
-Xlog:gc*:file=/var/log/app/gc.log:time,uptime,level,tags:filecount=10,filesize=100M

# === Logs de classes (utile pour debug ClassLoader leak) ===
-Xlog:class+load=info:file=/var/log/app/classloading.log

# === Diagnostic toujours actif ===
-XX:+UnlockDiagnosticVMOptions
-XX:+PrintFlagsFinal              # voir tous les flags effectifs (à mettre temporairement)
-XX:+PreserveFramePointer         # nécessaire pour async-profiler

# === Container awareness (JVM 10+) ===
-XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0

# === Code cache (rare mais utile sur grosse appli) ===
-XX:ReservedCodeCacheSize=512m

# === Crash dump ===
-XX:ErrorFile=/var/dumps/hs_err_pid%p.log

# === Network (timeout DNS, propagation cluster) ===
-Djava.net.preferIPv4Stack=true
-Dsun.net.inetaddr.ttl=60         # cache DNS plus court (défaut = forever)

    Java Agents · profilage et observabilité

    Java Flight Recorder (JFR)

    Profiler officiel intégré à la JVM. Quasi zéro overhead (< 1%), utilisable en production.

    # Démarrer un enregistrement de 60 secondes
jcmd <PID> JFR.start name=incident duration=60s filename=/tmp/incident.jfr

# Voir les enregistrements actifs
jcmd <PID> JFR.check

# Stopper et flusher
jcmd <PID> JFR.stop name=incident
jcmd <PID> JFR.dump name=incident filename=/tmp/incident.jfr

    Analyse offline : JDK Mission Control (JMC). Vues complètes · CPU samples, allocations par classe, contention de threads, I/O, etc.

    async-profiler

    Profiler natif basé sur perf, plus précis que JFR sur les CPU samples. Très utilisé en prod.

    # Profile CPU pendant 30s, génère un flamegraph SVG
./profiler.sh -d 30 -f /tmp/cpu.svg <PID>

# Profile allocations
./profiler.sh -d 30 -e alloc -f /tmp/alloc.svg <PID>

# Profile lock contention
./profiler.sh -d 30 -e lock -f /tmp/lock.svg <PID>

    Le flamegraph : visualisation où la largeur des barres = temps CPU passé. Un bloc large en haut = méthode chaude. Indispensable pour optimiser.

    Byte Buddy / ASM

    Pour les agents custom (instrumentation à la volée). Utilisé par tous les APM modernes (Datadog, Dynatrace, AppDynamics) pour s'attacher sans modifier le code.

    Outils d'analyse offline

    Eclipse MAT (Memory Analyzer Tool)

    Le standard pour analyser un heap dump (.hprof). Vues clés :

    Vue À quoi ça sert
    Leak Suspects Report MAT identifie automatiquement les objets retenus de manière suspecte
    Dominator Tree Qui retient le plus de mémoire (vue agrégée par sous-arbre)
    Histogram Classes triées par nombre d'instances ou bytes occupés
    OQL (Object Query Language) Requête SQL-like sur le heap (SELECT * FROM java.lang.String WHERE s.value.length > 1000)
    Path to GC Roots Pourquoi un objet n'est pas collecté (chaîne de références)

    VisualVM

    GUI tout-en-un (gratuit) : monitoring temps réel, sampling CPU, thread inspector, heap dump basique.

    JDK Mission Control (JMC)

    Compagnon de JFR. Vues très riches pour analyser un enregistrement de prod : événements GC, allocations, threads, I/O, exceptions, latency.

    IntelliJ Profiler

    Intégré directement dans IntelliJ Ultimate. Pratique pour profiler local + lire les flamegraphs.

    Cas concrets

    Comment détecter une fuite mémoire ?

    Signal classique : la Old generation (jstat -gcutil colonne O) ne redescend jamais après les Full GC. Elle croît lentement jusqu'à l'OOM. Méthode : (1) activer -XX:+HeapDumpOnOutOfMemoryError, (2) attendre l'OOM ou forcer un dump avec jmap -dump:live, (3) ouvrir dans MAT, (4) lire le Leak Suspects Report, (5) suivre le Path to GC Roots de l'objet suspect.

    Quelle différence entre Minor GC, Major GC et Full GC ?

    Minor GC : nettoie seulement la Young generation. Très rapide (ms). Major GC : nettoie la Old generation. Plus long. Full GC : nettoie TOUT (Young + Old + Metaspace). C'est le plus coûteux, à minimiser absolument en production.

    Pourquoi Xms = Xmx en production ?

    Sans ça, la JVM démarre avec une heap petite et la grandit à chaud quand nécessaire. Chaque agrandissement = pause + risque de fragmentation. En prod, on alloue dès le départ le maximum pour éviter ce coût.

    G1 ou ZGC pour une plateforme de paiement ?

    G1 pour 99% des cas : par défaut depuis Java 9, pauses prévisibles (~200ms), heap jusqu'à 32 Go bien géré. ZGC si la latence p99 doit rester < 10ms quelle que soit la taille du heap (jusqu'au To). Le surcoût RAM (~10-15%) est le compromis.

    Comment lire un thread dump efficacement ?

    Faire 3 dumps espacés de 10s avec jstack. Comparer : un thread BLOCKED au même endroit dans les 3 dumps = contention forte sur un lock. Chercher socketRead0 (appel réseau bloquant), getConnection (pool DB saturé), synchronized répété (lock contention). Le mot-clé Found one Java-level deadlock en bas du dump = deadlock détecté par la JVM.

    Quand passer aux Virtual Threads ?

    Si l'appli est bloquante par nature (appels DB, HTTP synchrones vers des banques acquéreurs) : passer aux virtual threads Java 21 retire le besoin de gros pools de threads physiques et améliore drastiquement le scaling sans rewrite. Sur du CPU-bound ou full async (WebFlux), pas de gain.

    Production en conteneur, quoi de spécial ?

    Activer -XX:+UseContainerSupport (défaut JVM 10+), utiliser -XX:MaxRAMPercentage plutôt que -Xmx (calcul automatique en fonction des limites cgroup), monter le dump path sur un volume persistant (-XX:HeapDumpPath=/dumps) sinon perdu au crash, exporter les métriques JVM via Micrometer + Prometheus.

    Points clés

    • La JVM = heap (Xms/Xmx) + Metaspace + Code Cache + Thread stacks + Off-heap. Le RSS dépasse toujours le Xmx
    • G1GC par défaut en prod • ZGC quand latence p99 critique avec gros heap
    • -Xms = -Xmx toujours en production
    • Logs GC actifs en permanence (-Xlog:gc*)
    • Auto-dump sur OOM (-XX:+HeapDumpOnOutOfMemoryError) + -XX:+ExitOnOutOfMemoryError
    • Les 8 types d'OOM ont des causes différentes • toujours lire le message complet
    • 3 thread dumps espacés > 1 thread dump isolé
    • JFR + async-profiler = la combinaison gagnante pour profiler en prod
    • Virtual Threads (Java 21) = game changer sur workloads bloquants
