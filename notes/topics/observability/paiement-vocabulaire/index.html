    3ds architecture iso8583 monétique paiement vocabulaire

    Vocabulaire & Architecture du Paiement

    Cette page rassemble les notions métier indispensables pour parler crédiblement à un acteur du paiement : qui sont les acteurs, comment circule l'argent, quels sont les standards et statuts d'une transaction.

    C'est l'angle qui te manque si tu viens du dev pur : tu peux maîtriser Kafka, mais si tu confonds autorisation et capture, acquéreur et issuer, tu perds la confiance d'un manager monétique en 30 secondes.

    Les acteurs · qui fait quoi

    ┌──────────┐                               ┌──────────┐
│ Acheteur │                               │  Issuer  │  (banque émettrice de la carte)
└────┬─────┘                               └────▲─────┘
     │                                          │
     │ paie avec carte                          │
     ▼                                          │
┌──────────┐    ┌──────────┐    ┌──────────┐   │
│ Marchand │───▶│ Gateway  │───▶│ Acquirer │───▶│  via Scheme (Visa / MC)
│ (e-com,  │    │   PSP    │    │ (banque  │
│  POS)    │    │          │    │  acquér.)│
└──────────┘    └──────────┘    └──────────┘
    Acteur Rôle Exemple
    Acheteur (cardholder) Possède la carte, paie Vous, moi
    Marchand (merchant) Vend, accepte les paiements E-commerçant, magasin physique
    Gateway / PSP • Payment Service Provider Interface technique entre marchand et acquéreur, gère TPE virtuels et redirections Stripe, Adyen, Worldline, Payline, Mangopay, Mollie, Lyra
    Acquéreur (acquiring bank) Banque qui héberge le compte marchand et reçoit les fonds BNP, Crédit Mutuel, Société Générale côté acquisition
    Scheme (réseau) Réseau international qui route les transactions entre acquéreur et émetteur Visa, Mastercard, Amex, JCB, UnionPay, CB (France)
    Issuer (issuing bank) Banque qui a émis la carte de l'acheteur, gère son compte Sa banque
    Régulateur Encadre l'activité Banque de France, ACPR, BCE, EBA

    À retenir : l'acquéreur et l'issuer ne se parlent jamais directement. Ils passent toujours par le scheme. C'est ce qui fait que Visa Mastercard sont aussi puissants : ils sont au centre du graphe.

    Le flux d'une transaction · 3 étapes distinctes

    C'est le point que confondent les non-paiement : transaction n'est pas un événement atomique, c'est un cycle en plusieurs étapes étalées dans le temps.

    1. Autorisation (Authorization)

    Question posée : la carte est-elle valide, le compte a-t-il les fonds, le risque est-il acceptable ?

    Marchand → PSP → Acquirer → Scheme → Issuer
                                         │
                                         ▼
                                  vérifs : plafond, fonds,
                                  fraude, 3DS, carte non bloquée
                                         │
                                         ▼
Marchand ◀ PSP ◀ Acquirer ◀ Scheme ◀──── OK / KO + code réponse
    • Durée typique : 300 ms à 3 secondes
    • Résultat : code réponse (00 = OK, 51 = NSF insufficient funds, 14 = invalid card, 05 = do not honor, etc.)
    • Effet : les fonds sont réservés sur la carte (hold) mais pas encore transférés

    2. Capture (Capture / Settlement Request)

    Question posée : on confirme qu'on veut bien encaisser, on déclenche le mouvement de fonds.

    • Peut être immédiate après autorisation (capture automatique) ou différée (typique e-commerce : capture à l'expédition)
    • Plusieurs captures possibles sur une seule autorisation (split capture)
    • Une autorisation non capturée expire après quelques jours (3 à 30 selon scheme et type)

    3. Settlement (compensation)

    Question posée : transfert effectif d'argent entre banques.

    • Batché en fin de journée (clearing)
    • L'acquéreur reçoit les fonds de l'issuer via le scheme
    • L'acquéreur reverse au marchand (J+1 à J+5 typique selon contrat)
    • Frais déduits : interchange (issuer), scheme fee, acquirer fee, PSP markup
    J+0    Authorization + Capture
J+0/J+1 Clearing (batch nocturne)
J+1    Settlement scheme → acquéreur
J+1/J+5 Versement marchand (selon contrat PSP)

    Statuts d'une transaction · cycle de vie complet

    Statut Signification Cas
    INITIATED Demande créée côté PSP, pas envoyée Étape interne
    PENDING En attente de réponse (3DS, banque) Asynchrone
    AUTHORIZED Banque a autorisé, fonds réservés OK technique, pas encore encaissé
    DECLINED Banque a refusé Code 51, 14, 05...
    CAPTURED Capture demandée par le marchand Fonds en cours de transfert
    SETTLED Compensation effective Marchand a touché les fonds
    REFUNDED Remboursement complet Initié par le marchand
    PARTIALLY_REFUNDED Remboursement partiel
    REVERSED Autorisation annulée avant capture Annulation rapide
    CHARGEBACK Le porteur conteste, scheme tranche Recours acheteur
    EXPIRED Autorisation non capturée à temps Timeout métier

    3-D Secure (3DS) · l'authentification forte

    Protocole d'authentification du porteur pendant la transaction. Évolution :

    Version Caractéristique
    3DS 1 (déprécié) Redirection vers ACS de la banque, OTP par SMS ou mot de passe statique
    3DS 2 (actuel) Fluide, mobile-friendly, frictionless flow si risque faible (RBA · Risk-Based Authentication)

    Flow 3DS 2

    Marchand → PSP → 3DS Server → DS (Directory Server scheme) → ACS (Access Control Server issuer)
                                                                  │
                                                                  ▼
                                                          analyse risque
                                                                  │
                                ┌─────────────────────────────────┤
                                ▼                                 ▼
                       frictionless (90%)              challenge (10%)
                                │                                 │
                                │                                 ▼
                                │                          OTP / biométrie
                                │                                 │
                                ▼                                 ▼
                                  authentication successful
                                                │
                                                ▼
                                          autorisation classique

    SCA (Strong Customer Authentication)

    Imposée par la DSP2 (Directive sur les Services de Paiement 2) en Europe pour la majorité des transactions e-commerce. Repose sur 2 facteurs parmi 3 :

    1. Knowledge : ce que je sais (mot de passe, code)
    2. Possession : ce que je possède (téléphone, carte)
    3. Inherence : ce que je suis (empreinte, faciale)

    Exemptions SCA

    Cas où SCA peut être évitée : - Low Value Transaction : < 30 € - Recurring Transaction : abonnement après 1re SCA - MIT • Merchant Initiated Transaction (déclenchée par le marchand sans porteur présent) - TRA • Transaction Risk Analysis : si le risque calculé est faible

    Standards · ISO 8583 et ISO 20022

    ISO 8583

    Le standard historique des messages de cartes de paiement (1987, révisions 1993, 2003). Format binaire ou hex, peu lisible mais ultra-compact.

    Type message : 0100 (autorisation)
Bitmap : indique quels champs sont présents
Données :
  Field 2  : PAN
  Field 3  : Processing code
  Field 4  : Amount
  Field 7  : Transmission date/time
  Field 11 : System trace audit number (STAN)
  Field 38 : Authorization code
  Field 39 : Response code (00 = OK)
  Field 41 : Terminal ID
  Field 42 : Merchant ID
  ...

    À connaître : MTI (Message Type Indicator) 0100 = request, 0110 = response, 0200 = financial, 0420 = reversal, etc. Code réponse en field 39.

    ISO 20022

    Standard moderne XML / JSON pour tous types de paiements (cartes, virements, prélèvements). Adopté progressivement par les schemes (Visa, Mastercard) et SWIFT.

    • Format structuré, autodocumenté
    • Universel (cartes, SEPA, SWIFT migrent dessus)
    • Coexistera avec ISO 8583 pendant 10+ ans

    Types de paiements (au-delà de la carte)

    Type Description Particularité
    Carte Visa, MC, Amex, CB Le plus courant en e-com, complexe
    SEPA Credit Transfer (SCT) Virement J+1 ouvré, gratuit, irrévocable
    SEPA Instant (SCT Inst) Virement instantané < 10s, 24/7, plafond 100k €
    SEPA Direct Debit (SDD) Prélèvement Initié par le créancier, mandat requis
    Wallet Apple Pay, Google Pay, Paypal Routent in fine sur carte ou compte
    BNPL • Buy Now Pay Later Klarna, Alma, Floa Mini-crédit court terme
    PIS • Payment Initiation Service Bridge, Tink, Powens DSP2, déclenche un virement bancaire depuis le compte client
    Crypto BTC, USDC, etc. Souvent via stablecoin pour réduire la volatilité

    Notions financières adjacentes à connaître

    Notion Signification
    Interchange Frais que l'acquéreur paie à l'issuer (~0,2-0,3 % en EU sur cartes consommateur)
    Scheme fee Frais du scheme (Visa, MC) à chaque transaction
    MDR • Merchant Discount Rate Frais total facturé au marchand par le PSP
    Chargeback Contestation par le porteur, le marchand peut perdre la somme + frais
    Reversal Annulation rapide d'une autorisation avant capture (côté technique)
    Refund Remboursement après capture (côté métier)
    Settlement window Délai entre capture et versement marchand
    Cut-off time Heure limite quotidienne pour qu'une capture entre dans le batch du jour
    Pre-authorization Réserve plus que prévu (hôtel, location de voiture) puis ajuste à la capture
    Recurring Paiement répété sur la même carte (abonnement)
    Card-on-file (CoF) Carte enregistrée chez le marchand pour usage ultérieur (tokenisée)

    Codes de réponse usuels (ISO 8583 field 39)

    Code Signification Action
    00 Approved OK, on capture
    01 Refer to issuer Refus, contacter banque (rare en e-com)
    04 Pick up card Carte volée/perdue, ne pas honorer
    05 Do not honor Refus générique (la plupart des refus)
    12 Invalid transaction Format de message incorrect
    14 Invalid card number PAN invalide
    30 Format error Erreur message
    41 Lost card Déclarée perdue
    43 Stolen card Déclarée volée
    51 Insufficient funds NSF
    54 Expired card Date d'expiration dépassée
    55 Invalid PIN PIN incorrect
    57 Transaction not permitted to cardholder Carte non autorisée pour ce type
    61 Exceeds withdrawal limit Plafond dépassé
    65 Activity count limit exceeded Trop de transactions sur la période
    91 Issuer or switch inoperative Indispo banque ou scheme
    96 System malfunction Indispo générique

    Cas concrets

    Quelle différence entre acquéreur et issuer ?

    Acquéreur = banque du marchand (acquisition des fonds). Issuer = banque émettrice de la carte du porteur. Sur une transaction, l'acquéreur acquiert les fonds que l'issuer émet. Ils communiquent via le scheme (Visa/MC).

    Pourquoi une autorisation peut-elle réussir et la transaction quand même échouer en capture ?

    Quelques cas : (1) la banque émettrice révoque l'autorisation entre les deux étapes (carte déclarée volée entre temps), (2) le marchand capture après expiration de l'autorisation (J+30 sur Visa), (3) le marchand capture un montant supérieur à l'autorisation initiale sans pré-autorisation. Toujours capturer rapidement et au bon montant.

    C'est quoi un chargeback ?

    Procédure de contestation initiée par le porteur via sa banque. Le scheme arbitre. Le marchand a un délai pour fournir des preuves (livraison signée, IP cohérente, 3DS authentifié...). Si perdu : perte du montant + frais de chargeback (~25 €). Au-delà d'un certain ratio de chargebacks, le marchand est mis en programme de monitoring scheme, voire blacklisté.

    Pourquoi 3DS frictionless plutôt que challenge ?

    Statistiquement, chaque challenge perd 30-40 % des transactions (abandons). Donc 3DS 2 a introduit un mode frictionless où l'issuer décide d'authentifier silencieusement si le risque est faible (RBA). Seul ~10 % des transactions vont en challenge réel. La SCA est respectée car le 3DS 2 utilise plusieurs facteurs d'analyse.

    MIT (Merchant Initiated Transaction), c'est quoi exactement ?

    Une transaction initiée par le marchand sans le porteur présent (abonnement mensuel, frais d'hôtel après checkout, ajustement de réservation). Doit s'appuyer sur une transaction initiale avec consentement explicite (CIT · Customer Initiated Transaction). En DSP2 : exempté de SCA si MIT déclarée correctement à l'autorisation initiale.

    ISO 8583 vs ISO 20022 : lequel doit-on connaître ?

    Les deux. ISO 8583 reste majoritaire sur les flux cartes legacy (échanges acquéreur/scheme/issuer). ISO 20022 monte fortement : SWIFT a migré (CBPR+ depuis 2022), Visa et Mastercard implémentent ISO 20022 sur certains flux. Sur les 5-10 prochaines années, coexistence avec passerelles.

    Comment identifier qui est responsable d'une lenteur sur une transaction ?

    En autorisation, la chaîne typique fait : Marchand → PSP → Acquirer → Scheme → Issuer → retour. Si on instrumente correctement, on a une trace par étape avec timestamps. Le coupable est généralement : l'issuer (sa banque ramassée) ou le scheme (pic de trafic Visa). Plus rarement l'acquéreur ou le PSP. Le marchand est rarement la cause s'il a une bonne stack moderne.

    Points clés

    • Acquéreur (banque marchand) et Issuer (banque porteur) ne se parlent qu'à travers le Scheme
    • Une transaction n'est pas atomique : Autorisation → Capture → Settlement, étalées sur plusieurs jours
    • 3DS 2 + SCA imposés par DSP2 en EU, avec exemptions (Low Value, MIT, recurring, TRA)
    • ISO 8583 (historique binaire) cohabite avec ISO 20022 (XML/JSON moderne)
    • Codes réponse à connaître : 00 (OK), 05 (refus), 14 (PAN invalide), 51 (NSF), 91 (banque indispo)
    • Chargeback = contestation par le porteur, peut coûter cher au marchand
    • MIT vs CIT = qui initie ? Impact direct sur les règles SCA
