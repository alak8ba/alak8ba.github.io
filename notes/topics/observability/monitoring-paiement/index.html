    Monitoring d'une Plateforme de Paiement

    Une plateforme monétique a des contraintes spécifiques qu'un monitoring générique ne couvre pas. Cette page applique les principes du RUN / MCO / SRE au domaine paiement temps réel, avec un focus sur le monitoring métier et le consumer lag Kafka.

    Pourquoi le monitoring paiement est différent

    Sur une appli SaaS classique, une indispo de 5 minutes est inconfortable. Sur une plateforme de paiement, c'est :

    • des millions d'euros de transactions non traitées
    • des marchands en panne sans alternative
    • des engagements contractuels SLA pénalisés
    • une perte de confiance difficile à reconquérir

    Conséquence : disponibilité quasi-totale exigée (99.95% à 99.99%), monitoring temps réel obligatoire, incident management ultra-rapide (P1 = traitement immédiat 24/7).

    MCO vs MCS

    Sur un système critique on distingue deux périmètres complémentaires :

    • MCO • Maintien en Condition Opérationnelle → garder le service en état de fonctionnement (perfs, dispo, incidents)
    • MCS • Maintien en Condition de Sécurité → garder le niveau de sécurité (patchs, CVE, conformité PCI DSS, certifications)

    Sur une plateforme de paiement, le MCS est aussi critique que le MCO : une vulnérabilité non patchée peut coûter plus cher qu'une indisponibilité.

    Les 3 niveaux appliqués au paiement

    Niveau infra

    Standard : CPU, RAM, heap JVM, disque, réseau. Rien de spécifique au paiement, mais avec des seuils plus stricts (alerte à 70% au lieu de 85%, marge de sécurité).

    Niveau applicatif

    Indicateur Pourquoi c'est critique en paiement
    Temps de réponse p99 Une autorisation > 3s = timeout côté marchand
    Threads Tomcat occupés Si saturation, les nouvelles transactions sont rejetées
    Pool connexions Oracle Saturation = blocage en cascade
    Heap Java + GC pauses Stop-the-world > 1s = chute de débit visible
    Latence vers banques acquéreurs Critère métier intégré : si > seuil, alerte

    Niveau métier • le plus important

    C'est ce qui distingue un bon responsable d'applications monétiques d'un simple ingénieur d'exploitation. Les indicateurs métier en paiement :

    Indicateur Définition Seuil d'alerte typique
    TPS • Transactions per second Débit instantané Chute > 30% sur 5 min
    Taux d'autorisation % de transactions acceptées par les banques Baisse > 5 points
    Taux de refus % refusées (motifs variés) Pic anormal
    Taux d'échec technique % timeout, erreurs réseau > 1% sustained
    Latence end-to-end Du POST marchand à la réponse p99 > 2s
    Volume par banque acquéreur Ventilation par partenaire Chute brutale d'une banque
    Volume par marchand top-N Détection d'une fraude ou panne marchand Pic ou chute anormal

    Un système peut être vert sur les niveaux 1 et 2 et rouge sur le niveau 3. Exemple : tous les serveurs répondent en 50ms, mais 100% des autorisations vers une banque acquéreur échouent depuis 8 minutes. C'est cette alerte que doit voir le RUN en premier.

    Stack technique typique

    Application Java (Tomcat, Spring) → Micrometer
                                       ↓
                                  Prometheus (scraping toutes les 15s)
                                       ↓
                                  Grafana (dashboards temps réel)
                                       ↓
                                  Alertmanager → astreinte (PagerDuty, OpsGenie)

Logs (catalina.out, applicatifs) → Filebeat/Promtail → Elasticsearch/Loki → Kibana/Grafana

    C'est exactement la stack standard de l'industrie. La spécificité paiement n'est pas dans les outils mais dans les métriques qu'on expose et les seuils qu'on définit.

    Investigation pas à pas • la boîte à outils du RUN

    C'est le cœur du métier. Quand une alerte tombe à 3h du matin, on suit un parcours toujours dans le même ordre : du bas (infra) vers le haut (métier). Voici les commandes et les signaux à connaître à chaque étage.

    Triage initial (30 secondes)

    Avant tout, situer l'incident :

    # Heure de l'alerte, heure du dernier déploiement
# → si écart < 30 min, suspect numéro 1 : la livraison

# Charge instantanée et nombre de processus
uptime
# 14:23:01 up 47 days, load average: 18.45, 12.10, 6.22
#                      ↑ très élevé = saturation CPU ou I/O

# Sessions actives sur le serveur
who

    uptime donne 3 chiffres : charge 1min / 5min / 15min. Si le 1min est très supérieur au 15min, l'incident est récent et brutal.

    Niveau 1 • Infrastructure (Linux)

    CPU

    # Vue temps réel par processus
top -c
# Touche P : tri par %CPU, M : tri par %MEM, H : voir threads

# Plus moderne et lisible
htop

# Détail par CPU et par état (user, system, iowait, idle)
mpstat -P ALL 2 5
# %iowait élevé → le CPU attend les disques ou le réseau, pas un vrai problème CPU

# Top des processus consommateurs (alternative ps)
pidstat -u 2 5
ps aux --sort=-%cpu | head -10

    Comprendre le Load Average · le chiffre le plus mal interprété de Linux :

    load average: 2.15, 1.98, 1.85   (moyennes sur 1, 5, 15 min)

    C'est le nombre moyen de processus en cours d'exécution OU en attente d'une ressource (CPU, I/O). À comparer au nombre de vCPU de la machine :

    Machine Load 1 Load 4 Load 8
    1 vCPU 100% utilisé saturé × 4 saturé × 8
    4 vCPU 25% utilisé 100% pile poil saturé × 2
    8 vCPU 12% utilisé 50% utilisé 100% utilisé

    Règle pratique : load > nb cores → saturation, load > nb cores × 2 → urgence.

    Signaux à connaître :

    Pattern Interprétation
    %user > 80% sustained L'appli boucle ou GC en pause
    %system > 30% Trop d'appels système, problème kernel ou I/O
    %iowait > 20% Disque ou réseau saturé, pas le CPU
    %steal > 5% (VM) L'hyperviseur prive la VM de CPU
    load > nb cores × 2 Saturation, processus en file d'attente

    Mémoire

    # Vue globale (lire la ligne "available", pas "free" - Linux mythe classique)
free -h
#               total   used    free    shared  buff/cache  available
# Mem:           32Gi   18Gi   1.2Gi    340Mi      12Gi        13Gi
#                                                              ↑ ce qui compte

# Détail par processus
ps aux --sort=-%mem | head -10

# Détails kernel (caches, slabs, swappable)
cat /proc/meminfo | head -20
# MemTotal, MemAvailable, SwapTotal, SwapFree, Cached, Buffers...

# Suivi temps réel
vmstat 2 10
# Colonne si/so : swap-in/swap-out. Si > 0 sustained = manque de RAM réel

    Réflexes : - swap qui s'active = RAM saturée, performances en chute libre - Heap Java qui dépasse Xmx → impossible (limite logicielle), mais le RSS du process peut dépasser de 30% à cause des structures off-heap (DirectByteBuffer, Metaspace, threads) - Lire available (vraie RAM dispo) et pas free (mythe Linux : le cache pris pour de l'occupé)

    Disque

    # Espace disque par filesystem
df -h
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/sda1        50G   35G   15G  70% /
# /dev/sda2         8G    1G    7G  12% [SWAP]
# /dev/sda3       512M   50M  462M  10% /boot/efi
# Alerter dès 80%, critique à 95%

# Plus gros consommateurs sous un répertoire
du -sh /var/log/* | sort -h

# I/O temps réel par device et par processus
iostat -xz 2 5
iotop -oP

# Fichiers ouverts par un PID (logs ouverts en lecture seule = OK,
# logs ouverts en écriture par dizaines = leak)
lsof -p <PID> | wc -l

    Comprendre les partitions Linux typiques :

    Mount Rôle Volumétrie Risques si plein
    / (root) Système, applications, logs par défaut 20-100 Go selon machine DB qui crashe, écritures qui échouent partout
    [SWAP] Extension disque de la RAM Souvent = RAM physique Si très utilisée = pression RAM grave
    /boot/efi Démarrage UEFI 100-500 Mo Empêche le prochain reboot
    /var Logs, caches, données apps Souvent dédié Logs qui saturent → appli muette
    /data Données métier (DB, Kafka) Dédié, gros volume Impact direct sur le service

    Sur une plateforme de paiement, séparer /var/log du / est une bonne pratique : si les logs explosent, le système continue à tourner.

    Signaux clés : - await (iostat) > 50ms sur un SSD = disque saturé - %util > 80% = device saturé - Filesystem plein = écritures qui échouent, base de données qui plante - Swap utilisé > 0 sustained = RAM saturée, vérifier en priorité avant le disque

    Réseau

    # Connexions actives - filtrer sur le port applicatif
ss -tnp state established '( dport = :8080 or sport = :8080 )'

# Connexions en TIME_WAIT (fermées récemment)
ss -tan | awk '{print $1}' | sort | uniq -c
# Trop de TIME_WAIT = épuisement des ports éphémères

# Statistiques par interface
ip -s link show eth0

# Bande passante temps réel par flux et IP
iftop -i eth0
# Vue interactive : qui consomme le réseau en ce moment

# Bande passante temps réel agrégée (entrée/sortie)
nload eth0

# Latence et perte vers une banque acquéreur
ping -c 10 acquirer.bank.com
mtr -rwc 50 acquirer.bank.com

# Capturer un trafic spécifique (incident)
tcpdump -i eth0 -w /tmp/capture.pcap host acquirer.bank.com and port 443

    Mesure de débit avec iperf3 · l'outil de référence pour valider si un goulot d'étranglement vient du réseau :

    # Sur la machine cible (serveur)
iperf3 -s
# Écoute sur le port 5201 TCP par défaut

# Depuis la machine source (client)
iperf3 -c <ip-serveur>                # test TCP 10s
iperf3 -c <ip-serveur> -t 30          # test 30 secondes
iperf3 -c <ip-serveur> --bidir        # bidirectionnel (émission + réception)
iperf3 -c <ip-serveur> -u -b 100M     # UDP à 100 Mbit/s (mesure pertes + gigue)
iperf3 -c <ip-serveur> -P 4           # 4 flux parallèles

# Lecture résultat
# [  5]  0.00-10.00 sec  1.10 GBytes  945 Mbits/sec
#                                      ↑ débit moyen mesuré

    Cas d'usage typique : un appli paie est lente, on suspecte le réseau vers la DB Oracle. On lance iperf3 entre l'appli et Oracle → si on obtient les 1 Gbit/s attendu, le réseau est innocent, le bug est dans l'appli ou la DB.

    Patterns à reconnaître : - Beaucoup de connexions CLOSE_WAIT = côté local on n'a pas fermé → bug applicatif - Beaucoup de SYN_RECV = SYN flood ou backlog saturé - RTT en augmentation vers la banque = problème réseau côté partenaire

    Niveau 2 • JVM et Application Java

    Identifier le processus

    # PID de tous les processus Java
jps -lv
# 12345  org.apache.catalina.startup.Bootstrap  -Xmx8g -Xms8g ...

# PID via pgrep
pgrep -f tomcat

    Santé de la JVM

    # GC stats temps réel (toutes les 2 secondes, 20 fois)
jstat -gc <PID> 2000 20
#  S0C    S1C    S0U    S1U    EC      EU      OC       OU      MC      MU       YGC YGCT FGC FGCT GCT
# 8192   8192   0      4500   65536   42000   524288  410000   54000   52000    245 12.3 5   8.7  21.0

# Colonnes clés :
# - OC/OU : Old generation Capacity / Used. Si OU/OC > 80% sustained → fuite mémoire
# - YGC/FGC : nombre de jeunes / full GC depuis le démarrage
# - FGCT : temps total passé en Full GC. Si croît vite → pression GC

# Variante en pourcentages (lecture plus rapide)
jstat -gcutil <PID> 5s
#   S0     S1     E      O      M     CCS    YGC    YGCT    FGC    FGCT     GCT
#  0.00  56.50  82.14  72.31  98.21  93.45    245   12.301      5    8.752  21.053
# Si la colonne O dépasse 80% durablement après un Full GC = fuite mémoire confirmée

# Synthèse JVM
jcmd <PID> VM.info
jcmd <PID> VM.uptime
jcmd <PID> VM.system_properties

# Vue heap rapide (alternative à jstat)
jcmd <PID> GC.heap_info
# garbage-first heap   total 8388608K, used 4123456K
# region size 8192K, 503 young (4120576K), 0 survivors (0K)
# Metaspace       used 52345K, capacity 53120K

    Thread dump • LA technique pour appli qui rame

    # Méthode 1 : signal SIGQUIT (le PID Java reçoit le signal)
kill -3 <PID>
# Le dump apparaît dans catalina.out (Tomcat) ou dans stdout

# Méthode 2 : jstack (préférée, capture ciblée)
jstack -l <PID> > /tmp/threaddump-$(date +%s).txt

# Méthode 3 : via jcmd
jcmd <PID> Thread.print > /tmp/threaddump-$(date +%s).txt

# RÈGLE D'OR : faire 3 dumps espacés de 10 secondes
# Permet de voir si un thread reste BLOCKED au même endroit (deadlock probable)
for i in 1 2 3; do
  jstack -l <PID> > /tmp/dump-$i.txt
  sleep 10
done

    Lecture d'un thread dump · états à reconnaître :

    État Signification Action
    RUNNABLE Le thread tourne Normal, sauf si pris en boucle infinie (CPU à 100%)
    BLOCKED (on lock) Attend un synchronized Si même lock dans 2 dumps successifs → contention forte
    WAITING (parking) Attend une notification Souvent normal (pool de threads en attente de tâche)
    TIMED_WAITING Attend avec timeout (sleep, future) Souvent un appel externe (HTTP, DB) en cours
    Deadlock détecté JVM signale explicitement Found one Java-level deadlock en bas du dump

    Mots-clés à rechercher dans un dump : - synchronized + at ... répété sur N threads → contention - socketRead0 (java.net) → bloqué sur un appel réseau (banque qui timeout) - getConnection (Hikari/JDBC) → saturation pool DB - GC task thread actif tout le temps → pression GC

    Heap dump • analyse mémoire / OOM

    # Heap dump à la demande (pendant que la JVM tourne)
jcmd <PID> GC.heap_dump /tmp/heap.hprof

# Forcer un Full GC avant pour ne capturer que ce qui survit
jcmd <PID> GC.run
jcmd <PID> GC.heap_dump /tmp/heap.hprof

# Alternative avec jmap (force un Full GC + ne garde que les objets vivants)
jmap -dump:live,format=b,file=/tmp/heap.hprof <PID>

# Auto-dump sur OutOfMemoryError (à activer dans les JVM args)
# -XX:+HeapDumpOnOutOfMemoryError
# -XX:HeapDumpPath=/var/dumps/

# Vue rapide histogramme (sans dump complet)
jcmd <PID> GC.class_histogram | head -30
#  num     #instances         #bytes  class name
#    1:        4 521 234     362 098 720  byte[]
#    2:        3 211 050     205 467 200  java.util.HashMap$Node

    Analyse offline : ouvrir le .hprof dans Eclipse MAT ou VisualVM. - Vue Leak Suspects : MAT identifie les objets retenus de manière suspecte - Vue Dominator Tree : qui retient le plus de mémoire - Vue Histogram : compter les instances par classe

    Logs GC (à activer dans toute prod)

    # JVM 11+ : flags à mettre au démarrage
-Xlog:gc*:file=/var/log/app/gc.log:time,uptime,level,tags:filecount=10,filesize=100M

# Analyser visuellement
# Outils gratuits : GCViewer, GCEasy.io (upload du fichier)

    À surveiller : - Durée des pauses Full GC (> 1 seconde = problème) - Fréquence : > 1 Full GC / minute = pression mémoire - Croissance de la Old generation après chaque GC = fuite probable

    Niveau 3 • Tomcat et serveur d'application

    # Logs principaux
tail -f /opt/tomcat/logs/catalina.out

# Logs d'accès (codes HTTP, latence)
tail -f /opt/tomcat/logs/localhost_access_log.*.txt
# Format typique : IP - - [date] "GET /api/auth HTTP/1.1" 200 142 1245
#                                                          ↑ taille ↑ durée ms

# Statistiques temps réel via JMX
# (Connecteur, Threads, ThreadPool)
jconsole <PID>    # GUI - ouvrir MBeans → Catalina → ThreadPool
# OU
jcmd <PID> ManagementAgent.start_local

# Tomcat Manager (interface web, à activer dans tomcat-users.xml)
# URL : http://host:8080/manager/status
# Vue : threads actifs, sessions, mémoire heap, déploiements

    APM (Application Performance Monitoring) · si l'entreprise dispose d'une solution, le diagnostic devient beaucoup plus rapide :

    APM Forces
    Dynatrace Auto-instrumentation, distributed tracing, IA d'analyse
    AppDynamics Cartographie des transactions, baseline auto, code-level
    Datadog APM Intégration native Datadog, dashboards puissants
    New Relic Vue end-to-end, BPM, RUM frontend
    Glowroot Open source, instrumentation Java légère

    Un APM remonte automatiquement les transactions lentes, les stack traces des erreurs et la carte des appels sortants (DB, banques acquéreurs). Plus besoin de plonger dans les thread dumps à chaque incident.

    Métriques Tomcat à surveiller : - currentThreadsBusy / maxThreads : si proche de 100%, on rejette les requêtes - connectionCount : connexions ouvertes en cours - requestProcessingTime : temps moyen par requête - errorCount : nombre de 5xx renvoyés

    # Compter les codes HTTP des 10 dernières minutes dans les access logs
awk '{print $9}' localhost_access_log.txt | sort | uniq -c | sort -rn
#   54321 200
#    1234 302
#     421 500   ← anormalement élevé ?
#      89 503

    Niveau 4 • Base de données et pool de connexions

    # Métriques du pool HikariCP exposées via Micrometer
# - hikaricp_connections_active : utilisées en ce moment
# - hikaricp_connections_idle : disponibles
# - hikaricp_connections_pending : threads qui attendent une connexion
# - hikaricp_connections_timeout : nombre de timeouts

# Si pending > 0 sustained → le pool est trop petit OU les queries sont trop lentes

    Côté Oracle (typique paiement) :

    -- Sessions actives en ce moment
SELECT s.sid, s.serial#, s.username, s.status, s.event, s.sql_id, s.seconds_in_wait
FROM v$session s
WHERE s.status = 'ACTIVE' AND s.username = 'PAYMENT_APP'
ORDER BY s.seconds_in_wait DESC;

-- Sessions bloquées (locks)
SELECT blocking_session, sid, serial#, wait_class, seconds_in_wait
FROM v$session
WHERE blocking_session IS NOT NULL;

-- Top queries lentes en cours
SELECT sql_id, executions, elapsed_time/1000000 AS sec, sql_text
FROM v$sql
WHERE last_active_time > SYSDATE - 1/24
ORDER BY elapsed_time DESC
FETCH FIRST 10 ROWS ONLY;

-- Plan d'exécution d'une query suspecte
EXPLAIN PLAN FOR <ma_requete>;
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

    Patterns d'incident DB classiques : - Pool saturé pendant une query lente : tous les threads de l'appli sont bloqués sur getConnection() (visible dans le thread dump) - Deadlock entre 2 sessions : l'une des deux est tuée par Oracle (ORA-00060) - Plan d'exécution qui change après une mise à jour de stats → query qui passe de 50ms à 30s - TNS-12516 / ORA-12519 : DB qui refuse les nouvelles connexions

    Niveau 5 • Dépendances externes (banques acquéreurs)

    # Le SSL handshake fonctionne-t-il ?
openssl s_client -connect acquirer.bank.com:443 -servername acquirer.bank.com </dev/null

# Quel certificat ? Expire-t-il ?
echo | openssl s_client -connect acquirer.bank.com:443 2>/dev/null \
  | openssl x509 -noout -dates -subject -issuer

# Temps de chaque étape d'une requête HTTPS
curl -w "\nDNS:    %{time_namelookup}s\nTCP:    %{time_connect}s\nSSL:    %{time_appconnect}s\nTotal:  %{time_total}s\nHTTP:   %{http_code}\n" \
  -o /dev/null -s https://acquirer.bank.com/health

# Latence depuis l'application (test depuis le conteneur)
docker exec payment-app curl -w "%{time_total}\n" -o /dev/null -s https://acquirer.bank.com/health

    Patterns d'incident côté partenaire : - DNS qui prend > 1s → souvent oublié, peut tuer une plateforme - Certificat expiré du partenaire (vu plus souvent qu'on ne le pense) - Réponse en 200 OK mais payload anormal (banque qui « répond » mais en mode dégradé)

    Niveau 6 • Métier (Grafana, Kafka, dashboards)

    C'est le niveau le plus important mais souvent le plus rapide à diagnostiquer parce que tout est déjà mesuré et visualisé :

    # Taux d'autorisation par minute
sum(rate(payment_authorizations_success_total[5m]))
  / sum(rate(payment_authorizations_total[5m]))

# Taux d'erreur par banque acquéreur
sum by (acquirer) (rate(payment_authorizations_error_total[5m]))
  / sum by (acquirer) (rate(payment_authorizations_total[5m]))

# Latence p99 end-to-end
histogram_quantile(0.99, rate(payment_e2e_duration_seconds_bucket[5m]))

# Consumer lag par groupe / topic
sum by (group, topic) (kafka_consumergroup_lag)

    Si le dashboard Grafana montre une chute du taux d'autorisation sur une seule banque et les autres niveaux sont verts, l'investigation est terminée : c'est un incident chez le partenaire. On bascule sur la procédure de communication acquéreur.

    Diagnostic typique • scénario complet

    Un cas d'école qui revient souvent en paiement et qui illustre tout le raisonnement infra → applicatif → métier :

    Un opérateur métier signale : « Le paiement est très lent depuis 10 minutes. »

    Procédure étape par étape :

    [ 1 ] Triage 30s
      → uptime : load 4 sur 8 vCPU → OK
      → dernier déploiement il y a 3 jours → pas de corrélation
      → time-window : depuis 14h10

[ 2 ] Infra
      → top / mpstat → CPU à 20 %                              OK ✅
      → free -h     → MemAvailable 12 Gi sur 32 Gi              OK ✅
      → df -h       → / à 65 %, /var à 70 %                     OK ✅
      → ss / mtr    → latence vers banques < 50 ms              OK ✅

      Conclusion partielle : le serveur va bien. Le problème est plus haut.

[ 3 ] Applicatif (Tomcat + JVM)
      → access logs : durée moyenne /paiement = 7 s          ALERTE ❌
      → jconsole / JMX : currentThreadsBusy = 195 / 200      ALERTE ❌
      → jstack ×3 : 180 threads BLOCKED sur
                    HikariPool.getConnection                ALERTE ❌

[ 4 ] Base de données
      → métriques Hikari : pool = 50 / 50 saturé             ALERTE ❌
      → v$session : 50 sessions ACTIVE sur Oracle
      → v$sql : query SELECT facture WHERE merchant_id = ?
                qui durait 30 ms hier dure 4 s aujourd'hui   ALERTE ❌
      → EXPLAIN PLAN : FULL TABLE SCAN
                       au lieu d'INDEX RANGE SCAN            ROOT CAUSE ⚠️

[ 5 ] Action
      → Hint /*+ INDEX(facture FACTURE_MERCHANT_IDX) */ en attendant
      → Ouverture ticket DBA pour rebuild stats sur la table FACTURE
      → Communication marchand : retour à la normale dans 10 min

[ 6 ] Postmortem
      → Pourquoi les stats Oracle ne sont plus à jour ?
      → Job DBMS_STATS auto-désactivé depuis la dernière migration
      → Action permanente : remettre le job + alerter sur sa désactivation

    Ce qu'il faut retenir : un serveur peut être vert sur tous les niveaux infra et le service être rouge en applicatif parce que tous les threads attendent une connexion DB. C'est l'exemple type qu'un manager de RUN attend du candidat : la capacité à descendre dans la stack jusqu'à la cause racine sans se laisser rassurer par les premiers tableaux de bord verts.

    Synthèse • parcours d'investigation type

    Alerte (latence p99 > 2s)
  │
  ▼
[1] uptime / top         → CPU à 95%, load 25 sur 8 cores
  │
  ▼
[2] jstat -gc            → Old gen à 92%, 5 Full GC en 30s
  │
  ▼
[3] jstack ×3            → 80 threads BLOCKED sur HikariPool.getConnection
  │
  ▼
[4] Dashboard DB         → 200 sessions actives, top SQL = SELECT lent
  │
  ▼
[5] EXPLAIN PLAN         → plan changé suite à statistiques DB (table sans index récent)
  │
  ▼
Action immédiate         → Rollback du plan via hint OR DBA reconstruit l'index
Postmortem               → Pourquoi le plan a changé ? Maj stats automatique non maîtrisée ?

    C'est ce déroulé du bas vers le haut, avec une commande concrète à chaque étape, qu'un responsable d'application maîtrise. À l'inverse, sauter directement au métier sans avoir vérifié l'infra fait perdre du temps si le vrai problème était plus bas.

    Consumer Lag Kafka • un super exemple de monitoring métier temps réel

    Le contexte

    Beaucoup de plateformes de paiement modernes utilisent Kafka pour découpler les flux : front d'acquisition → topic Kafka → consumer qui appelle la banque acquéreur → topic réponse → consumer qui notifie le marchand.

    Définition du consumer lag

    Le lag d'un consumer Kafka, c'est le nombre de messages produits mais pas encore lus par un consumer group sur un topic donné.

    Topic « auth-requests » :
    Partition 0 : offset producer = 12 543 091
                  offset consumer = 12 542 850
                  LAG = 241 messages

    Pourquoi c'est un indicateur métier crucial

    Cas Interprétation métier
    Lag stable proche de 0 Le système traite en temps réel, tout va bien
    Lag qui augmente lentement Le consumer ne suit plus le débit • risque de timeout marchand
    Lag qui explose Quelque chose bloque (banque acquéreur ralentie, pool DB saturé, GC pause)
    Lag qui redescend après pic Récupération en cours • capacité de rattrapage OK

    C'est un monitoring du flux métier, pas de la machine. Le serveur peut être à 20% de CPU et le lag être catastrophique parce que le consumer attend la banque.

    Comment l'exposer en Prometheus

    # Kafka Lag Exporter (Strimzi, kafka-exporter, Burrow…)
kafka_consumergroup_lag{group="auth-consumer", topic="auth-requests", partition="0"} 241

    Comment alerter

    # règle Prometheus
- alert: KafkaConsumerLagHigh
  expr: sum by (group, topic) (kafka_consumergroup_lag) > 5000
  for: 2m
  labels:
    severity: P1
  annotations:
    summary: "Consumer {{ $labels.group }} en retard de {{ $value }} messages"
    runbook: "https://runbook.internal/kafka-lag"

    Pourquoi c'est ton meilleur argument en entretien RUN

    Le consumer lag c'est exactement la mentalité monitoring du business plutôt que de la machine. C'est la même logique que surveiller un taux d'autorisation ou un volume de transactions par minute : on ne regarde pas "la machine va bien" mais "le flux métier est traité à temps".

    Si tu as exposé du Kafka lag avec Micrometer + Grafana sur un projet réel, tu peux dire "j'ai mis en place du monitoring temps réel orienté métier" · c'est exactement le vocabulaire qu'un responsable d'application monétique attend.

    Logs en contexte paiement

    Trois principes clés :

    1. Identifiant de transaction dans chaque ligne (trace ID, correlation ID, payment ID). Sans ça, on perd des heures pendant un incident.
    2. Structured logging (JSON) pour permettre des recherches efficaces : level=ERROR amount=42.5 acquirer=BNP duration_ms=2543
    3. Pas de données sensibles dans les logs (PAN, CVV, expiration) → conformité PCI DSS

    Centralisation typique : Filebeat / Promtail → Elasticsearch / Loki → Kibana / Grafana. Indispensable pour corréler 10 microservices pendant un incident.

    Gestion d'incident en paiement

    Sévérités typiques

    Sévérité Critère SLA interne
    P1 Service totalement indisponible OU > 50% transactions en échec Prise en compte < 15 min, communication client < 30 min
    P2 Une banque acquéreur down OU latence dégradée significative Prise en compte < 1h
    P3 Bug fonctionnel sans impact métier Prise en compte < 4h ouvrées

    Procédure d'astreinte

    Alerte Alertmanager → PagerDuty → SMS + appel
        ↓ (si non acquitté en 5 min)
   Escalade niveau 2
        ↓ (si non acquitté en 10 min)
   Escalade manager

    Boucle d'investigation

    Alerte → Dashboard Grafana → Identifier le niveau impacté (infra/appli/métier)
       → Lecture logs filtrés sur la fenêtre + ID transaction
       → Corréler avec déploiements récents (changelog)
       → Tester les dépendances externes (ping banques)
       → Si nécessaire : thread dump (kill -3 JVM), heap dump (jmap)
       → Rollback ou fix → Vérifier retour à la normale → Postmortem

    Conformité dans le monitoring paiement

    Norme Implication pour le monitoring
    PCI DSS Aucune donnée carte (PAN, CVV) dans les logs ou métriques
    RGPD Données personnelles : minimisation, durée de rétention bornée
    DSP2 Traçabilité des opérations d'authentification forte (SCA)

    Sur un dashboard ou un dump heap, tout ce qui est sensible doit être masqué ou exclu. C'est un réflexe à intégrer dès l'instrumentation.

    Cas concrets

    Pourquoi le monitoring métier prime sur le monitoring infra ?

    Parce qu'un système peut être vert techniquement (CPU bas, latence correcte) et défaillant métier (transactions vers une banque tombées à zéro). Le métier touche directement le chiffre d'affaires et les SLA client. C'est l'alerte qui doit déclencher en premier.

    Comment expliquer le consumer lag à un non-technique ?

    « C'est le retard accumulé entre ce qui rentre dans le système et ce qui en sort. Si le lag est de 5 000 messages, ça veut dire que 5 000 paiements sont en file d'attente et n'ont pas encore été traités. En temps réel, c'est inacceptable. »

    Quels outils pour exposer le consumer lag Kafka ?

    kafka-exporter (Burrow), Strimzi Kafka Exporter ou directement les métriques JMX exposées via Micrometer dans une appli Spring Boot. Ces métriques arrivent ensuite dans Prometheus et sont visualisées dans Grafana.

    Pourquoi l'identifiant de transaction dans les logs est si important ?

    Parce qu'une transaction passe par 5 à 10 microservices. Sans ID corrélé partout, retrouver toutes les traces d'une transaction qui a échoué peut prendre des heures. Avec un ID dans chaque log, une seule recherche Kibana donne tout le parcours.

    Quelle latence p99 viser sur une autorisation paiement ?

    Dépend du marchand et du flux, mais typiquement < 2 secondes end-to-end. Au-delà, le terminal de paiement ou la page e-commerce timeout. Le p99 est plus important que la moyenne : il révèle les cas pathologiques que la moyenne lisse.

    Thread dump et heap dump, quand les utiliser ?

    Thread dump (kill -3 ou jstack) : quand l'appli rame ou se fige, pour voir ce que font tous les threads (deadlock ? attente synchronized ? appel externe bloquant ?). Heap dump (jmap -dump) : quand on suspecte une fuite mémoire ou un OutOfMemoryError, pour voir ce qui occupe la heap. Outils d'analyse : Eclipse MAT, VisualVM.

    Points clés

    • Le monitoring paiement = stack standard, mais avec des seuils plus stricts et un monitoring métier obligatoire
    • Indicateurs métier = TPS, taux d'autorisation, taux de refus, latence end-to-end, volume par banque acquéreur
    • Le consumer lag Kafka est l'archétype du monitoring métier temps réel : on surveille le flux, pas la machine
    • ID de transaction corrélé dans tous les logs = indispensable pour l'investigation
    • P1 = traitement immédiat 24/7, astreinte avec escalade automatique
    • Conformité PCI DSS / RGPD = jamais de donnée sensible dans logs ou métriques
