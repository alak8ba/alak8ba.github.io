    Monitoring d'une Plateforme de Paiement

    Une plateforme monétique a des contraintes spécifiques qu'un monitoring générique ne couvre pas. Cette page applique les principes du RUN / MCO / SRE au domaine paiement temps réel, avec un focus sur le monitoring métier et le consumer lag Kafka.

    Pourquoi le monitoring paiement est différent

    Sur une appli SaaS classique, une indispo de 5 minutes est inconfortable. Sur une plateforme de paiement, c'est :

    • des millions d'euros de transactions non traitées
    • des marchands en panne sans alternative
    • des engagements contractuels SLA pénalisés
    • une perte de confiance difficile à reconquérir

    Conséquence : disponibilité quasi-totale exigée (99.95% à 99.99%), monitoring temps réel obligatoire, incident management ultra-rapide (P1 = traitement immédiat 24/7).

    Les 3 niveaux appliqués au paiement

    Niveau infra

    Standard : CPU, RAM, heap JVM, disque, réseau. Rien de spécifique au paiement, mais avec des seuils plus stricts (alerte à 70% au lieu de 85%, marge de sécurité).

    Niveau applicatif

    Indicateur Pourquoi c'est critique en paiement
    Temps de réponse p99 Une autorisation > 3s = timeout côté marchand
    Threads Tomcat occupés Si saturation, les nouvelles transactions sont rejetées
    Pool connexions Oracle Saturation = blocage en cascade
    Heap Java + GC pauses Stop-the-world > 1s = chute de débit visible
    Latence vers banques acquéreurs Critère métier intégré : si > seuil, alerte

    Niveau métier • le plus important

    C'est ce qui distingue un bon responsable d'applications monétiques d'un simple ingénieur d'exploitation. Les indicateurs métier en paiement :

    Indicateur Définition Seuil d'alerte typique
    TPS • Transactions per second Débit instantané Chute > 30% sur 5 min
    Taux d'autorisation % de transactions acceptées par les banques Baisse > 5 points
    Taux de refus % refusées (motifs variés) Pic anormal
    Taux d'échec technique % timeout, erreurs réseau > 1% sustained
    Latence end-to-end Du POST marchand à la réponse p99 > 2s
    Volume par banque acquéreur Ventilation par partenaire Chute brutale d'une banque
    Volume par marchand top-N Détection d'une fraude ou panne marchand Pic ou chute anormal

    Un système peut être vert sur les niveaux 1 et 2 et rouge sur le niveau 3. Exemple : tous les serveurs répondent en 50ms, mais 100% des autorisations vers une banque acquéreur échouent depuis 8 minutes. C'est cette alerte que doit voir le RUN en premier.

    Stack technique typique

    Application Java (Tomcat, Spring) → Micrometer
                                       ↓
                                  Prometheus (scraping toutes les 15s)
                                       ↓
                                  Grafana (dashboards temps réel)
                                       ↓
                                  Alertmanager → astreinte (PagerDuty, OpsGenie)

Logs (catalina.out, applicatifs) → Filebeat/Promtail → Elasticsearch/Loki → Kibana/Grafana

    C'est exactement la stack standard de l'industrie. La spécificité paiement n'est pas dans les outils mais dans les métriques qu'on expose et les seuils qu'on définit.

    Consumer Lag Kafka • un super exemple de monitoring métier temps réel

    Le contexte

    Beaucoup de plateformes de paiement modernes utilisent Kafka pour découpler les flux : front d'acquisition → topic Kafka → consumer qui appelle la banque acquéreur → topic réponse → consumer qui notifie le marchand.

    Définition du consumer lag

    Le lag d'un consumer Kafka, c'est le nombre de messages produits mais pas encore lus par un consumer group sur un topic donné.

    Topic « auth-requests » :
    Partition 0 : offset producer = 12 543 091
                  offset consumer = 12 542 850
                  LAG = 241 messages

    Pourquoi c'est un indicateur métier crucial

    Cas Interprétation métier
    Lag stable proche de 0 Le système traite en temps réel, tout va bien
    Lag qui augmente lentement Le consumer ne suit plus le débit • risque de timeout marchand
    Lag qui explose Quelque chose bloque (banque acquéreur ralentie, pool DB saturé, GC pause)
    Lag qui redescend après pic Récupération en cours • capacité de rattrapage OK

    C'est un monitoring du flux métier, pas de la machine. Le serveur peut être à 20% de CPU et le lag être catastrophique parce que le consumer attend la banque.

    Comment l'exposer en Prometheus

    # Kafka Lag Exporter (Strimzi, kafka-exporter, Burrow…)
kafka_consumergroup_lag{group="auth-consumer", topic="auth-requests", partition="0"} 241

    Comment alerter

    # règle Prometheus
- alert: KafkaConsumerLagHigh
  expr: sum by (group, topic) (kafka_consumergroup_lag) > 5000
  for: 2m
  labels:
    severity: P1
  annotations:
    summary: "Consumer {{ $labels.group }} en retard de {{ $value }} messages"
    runbook: "https://runbook.internal/kafka-lag"

    Pourquoi c'est ton meilleur argument en entretien RUN

    Le consumer lag c'est exactement la mentalité monitoring du business plutôt que de la machine. C'est la même logique que surveiller un taux d'autorisation ou un volume de transactions par minute : on ne regarde pas "la machine va bien" mais "le flux métier est traité à temps".

    Si tu as exposé du Kafka lag avec Micrometer + Grafana sur un projet réel, tu peux dire "j'ai mis en place du monitoring temps réel orienté métier" — c'est exactement le vocabulaire qu'un responsable d'application monétique attend.

    Logs en contexte paiement

    Trois principes clés :

    1. Identifiant de transaction dans chaque ligne (trace ID, correlation ID, payment ID). Sans ça, on perd des heures pendant un incident.
    2. Structured logging (JSON) pour permettre des recherches efficaces : level=ERROR amount=42.5 acquirer=BNP duration_ms=2543
    3. Pas de données sensibles dans les logs (PAN, CVV, expiration) → conformité PCI DSS

    Centralisation typique : Filebeat / Promtail → Elasticsearch / Loki → Kibana / Grafana. Indispensable pour corréler 10 microservices pendant un incident.

    Gestion d'incident en paiement

    Sévérités typiques

    Sévérité Critère SLA interne
    P1 Service totalement indisponible OU > 50% transactions en échec Prise en compte < 15 min, communication client < 30 min
    P2 Une banque acquéreur down OU latence dégradée significative Prise en compte < 1h
    P3 Bug fonctionnel sans impact métier Prise en compte < 4h ouvrées

    Procédure d'astreinte

    Alerte Alertmanager → PagerDuty → SMS + appel
        ↓ (si non acquitté en 5 min)
   Escalade niveau 2
        ↓ (si non acquitté en 10 min)
   Escalade manager

    Boucle d'investigation

    Alerte → Dashboard Grafana → Identifier le niveau impacté (infra/appli/métier)
       → Lecture logs filtrés sur la fenêtre + ID transaction
       → Corréler avec déploiements récents (changelog)
       → Tester les dépendances externes (ping banques)
       → Si nécessaire : thread dump (kill -3 JVM), heap dump (jmap)
       → Rollback ou fix → Vérifier retour à la normale → Postmortem

    Conformité dans le monitoring paiement

    Norme Implication pour le monitoring
    PCI DSS Aucune donnée carte (PAN, CVV) dans les logs ou métriques
    RGPD Données personnelles : minimisation, durée de rétention bornée
    DSP2 Traçabilité des opérations d'authentification forte (SCA)

    Sur un dashboard ou un dump heap, tout ce qui est sensible doit être masqué ou exclu. C'est un réflexe à intégrer dès l'instrumentation.

    Cas concrets

    Pourquoi le monitoring métier prime sur le monitoring infra ?

    Parce qu'un système peut être vert techniquement (CPU bas, latence correcte) et défaillant métier (transactions vers une banque tombées à zéro). Le métier touche directement le chiffre d'affaires et les SLA client. C'est l'alerte qui doit déclencher en premier.

    Comment expliquer le consumer lag à un non-technique ?

    « C'est le retard accumulé entre ce qui rentre dans le système et ce qui en sort. Si le lag est de 5 000 messages, ça veut dire que 5 000 paiements sont en file d'attente et n'ont pas encore été traités. En temps réel, c'est inacceptable. »

    Quels outils pour exposer le consumer lag Kafka ?

    kafka-exporter (Burrow), Strimzi Kafka Exporter ou directement les métriques JMX exposées via Micrometer dans une appli Spring Boot. Ces métriques arrivent ensuite dans Prometheus et sont visualisées dans Grafana.

    Pourquoi l'identifiant de transaction dans les logs est si important ?

    Parce qu'une transaction passe par 5 à 10 microservices. Sans ID corrélé partout, retrouver toutes les traces d'une transaction qui a échoué peut prendre des heures. Avec un ID dans chaque log, une seule recherche Kibana donne tout le parcours.

    Quelle latence p99 viser sur une autorisation paiement ?

    Dépend du marchand et du flux, mais typiquement < 2 secondes end-to-end. Au-delà, le terminal de paiement ou la page e-commerce timeout. Le p99 est plus important que la moyenne : il révèle les cas pathologiques que la moyenne lisse.

    Thread dump et heap dump, quand les utiliser ?

    Thread dump (kill -3 ou jstack) : quand l'appli rame ou se fige, pour voir ce que font tous les threads (deadlock ? attente synchronized ? appel externe bloquant ?). Heap dump (jmap -dump) : quand on suspecte une fuite mémoire ou un OutOfMemoryError, pour voir ce qui occupe la heap. Outils d'analyse : Eclipse MAT, VisualVM.

    Points clés

    • Le monitoring paiement = stack standard, mais avec des seuils plus stricts et un monitoring métier obligatoire
    • Indicateurs métier = TPS, taux d'autorisation, taux de refus, latence end-to-end, volume par banque acquéreur
    • Le consumer lag Kafka est l'archétype du monitoring métier temps réel : on surveille le flux, pas la machine
    • ID de transaction corrélé dans tous les logs = indispensable pour l'investigation
    • P1 = traitement immédiat 24/7, astreinte avec escalade automatique
    • Conformité PCI DSS / RGPD = jamais de donnée sensible dans logs ou métriques
