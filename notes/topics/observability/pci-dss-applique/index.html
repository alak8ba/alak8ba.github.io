    PCI DSS Appliqué · Ce qui change dans le code et l'exploitation

    PCI DSS (Payment Card Industry Data Security Standard) est le standard de sécurité imposé par les schemes (Visa, Mastercard, Amex, JCB, Discover) à tout acteur qui stocke, traite ou transmet des données de cartes de paiement. Pas une option · un prérequis légal pour opérer.

    Cette page se concentre sur ce que ça change concrètement côté dev, code, exploitation et monitoring · pas une lecture exhaustive de la norme.

    Les 12 grands principes

    # Principe (résumé) Côté code/run
    1 Firewall et segmentation réseau DMZ pour les composants exposés, isolation CDE
    2 Ne pas utiliser les configs par défaut (mots de passe, comptes) Hardening serveurs, comptes nommés
    3 Protéger les données de carte stockées Tokenization, encryption AES-256, masquage
    4 Chiffrer la transmission sur réseaux publics TLS 1.2+ obligatoire, pas de SSL, pas de TLS 1.0/1.1
    5 Antivirus / antimalware à jour EDR sur tous les serveurs
    6 Systèmes et apps sécurisés (CVE patching, secure dev) Dependabot, SCA, SAST, code reviews
    7 Restreindre l'accès aux données carte (besoin d'en connaître) RBAC strict, séparation des rôles
    8 Identifiant unique par utilisateur, MFA Pas de compte partagé, SSO + MFA
    9 Sécurité physique Datacenters certifiés
    10 Tracer tous les accès aux données carte Audit log immuable, centralisé, conservé 1 an
    11 Tester régulièrement la sécurité Pentest annuel, scans trimestriels
    12 Politique de sécurité formalisée Documentation, formation, plan incident

    Le concept central · CDE et tokenization

    CDE · Cardholder Data Environment

    C'est le périmètre qui touche aux données de carte. Tout ce qui : - stocke un PAN (numéro de carte) - traite un PAN - ou est connecté à un système qui en traite

    est dans le CDE et doit respecter PCI DSS. L'objectif d'architecture est de réduire le CDE au maximum.

    Tokenization

    Remplacer le PAN par un token sans valeur intrinsèque dès qu'il rentre dans le système.

    [Marchand]
  PAN 4111 1111 1111 1111
        ↓
  [Vault tokenization] (composant PCI DSS L1)
        ↓
  Token "TOK-a3f2e9...8b"
        ↓
  [Tout le reste du SI] → ne voit plus jamais le PAN

    Bénéfice : le reste du SI sort du CDE → réduction massive du périmètre audité.

    Données interdites de stockage

    Donnée Stockable ? Si oui, comment
    PAN (numéro complet) Oui mais évité Chiffré AES-256 ou masqué
    Cardholder name Oui Sans contrainte spécifique
    Expiration date Oui Sans contrainte spécifique
    Service code Oui Sans contrainte spécifique
    CVV / CVC / CID JAMAIS Interdit après autorisation
    PIN / PIN block JAMAIS Interdit après autorisation
    Full track data (magnétique/chip) JAMAIS Interdit après autorisation

    Règle d'or : CVV/CVC ne survit jamais à l'autorisation. Si tu en trouves un dans une DB, un log ou une trace : incident PCI DSS immédiat.

    Impact côté code applicatif

    Masquage à l'affichage

    public String maskPan(String pan) {
    // Format autorisé : 6 premiers + 4 derniers, le reste masqué
    if (pan == null || pan.length() < 13) return "****";
    return pan.substring(0, 6) + "******" + pan.substring(pan.length() - 4);
    // 411111******1111
}

    Encryption en base

    Préférer tokenization (vault dédié) à l'encryption applicative. Si encryption nécessaire :

    // AES-256-GCM avec clé en HSM ou KMS
@Convert(converter = PanEncryptedConverter.class)
@Column(name = "pan_encrypted")
private String pan;

public class PanEncryptedConverter implements AttributeConverter<String, byte[]> {
    @Override
    public byte[] convertToDatabaseColumn(String pan) {
        return kmsClient.encrypt("pci-pan-key", pan.getBytes());
    }
    @Override
    public String convertToEntityAttribute(byte[] cipher) {
        return new String(kmsClient.decrypt("pci-pan-key", cipher));
    }
}

    Jamais dans une URL

    ❌ GET /transactions?pan=4111111111111111
✅ POST /transactions  body: { "panToken": "TOK-a3f2..." }

    Les URLs finissent dans les access logs Nginx, Tomcat, navigateurs, proxies, monitoring URL · partout où elles ne devraient pas.

    Impact côté logs

    C'est le piège n°1 des équipes qui découvrent PCI DSS. Un dev en debug log son objet PaymentRequest toString → PAN en clair dans Splunk → audit fail.

    Règles d'or

    1. Aucun PAN complet dans les logs, même en niveau ERROR
    2. Aucun CVV nulle part, jamais
    3. Toujours masquer ou tokeniser avant le log
    4. toString() / Lombok @ToString : exclure les champs sensibles
    @ToString(exclude = {"pan", "cvv"})
public class PaymentRequest {
    private String pan;
    private String cvv;
    private BigDecimal amount;
}

    Ou mieux : ne jamais avoir cvv en mémoire après l'envoi à la banque.

    Logs autorisés et structure typique

    INFO  txId=tx-9281 merchant=42 amount=42.50 currency=EUR
      pan=411111******1111 status=AUTHORIZED auth_code=A1B2C3

    pan masqué selon BIN visible + 4 derniers. Suffit pour debug, conforme PCI DSS.

    Audit log spécifique (Requirement 10)

    Pour les accès aux données carte, log immuable dédié :

    ts=2026-06-04T14:32:11Z user=jdoe action=READ_PAN entity=txId-9281 reason=incident-INC-4321 ip=10.20.3.42

    Conservation minimum 1 an (3 mois en ligne, 9 mois archivé chiffré).

    Impact côté monitoring et observabilité

    Métriques Prometheus

    Ne jamais mettre une donnée carte dans un label. Les labels finissent en time series, accessibles via API Grafana, et restent stockées des mois.

    ❌ payment_authorizations_total{pan="411111******1111", merchant="42"} 1
✅ payment_authorizations_total{merchant="42", scheme="VISA", country="FR"} 1

    APM (Dynatrace / AppDynamics)

    Ces outils capturent automatiquement les valeurs de paramètres des méthodes (pour le distributed tracing). Risque : capture du PAN passé en paramètre.

    Solution : configurer le data masking de l'APM sur les méthodes qui touchent au PAN, ou marquer les classes avec une annotation que l'agent reconnaît.

    Heap dumps

    Un heap dump capture tout ce qui est en mémoire · y compris les objets String contenant des PAN, CVV, etc.

    Règles : - Chiffrer les heap dumps au moment de la génération - Stockage sur volume chiffré - Accès restreint et logué - Destruction après analyse

    # Heap dump direct sur volume chiffré, accès root only
-XX:HeapDumpPath=/secure-dumps/   # mount LUKS, mode 700

    Network · TLS et segmentation

    TLS strict

    # nginx.conf · PCI DSS-compliant
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-RSA-AES256-GCM-SHA384:ECDHE-RSA-CHACHA20-POLY1305;
ssl_prefer_server_ciphers on;
ssl_session_tickets off;
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload";

    Tester avec SSL Labs ou testssl.sh. Cible : A+ obligatoire.

    Segmentation

    CDE doit être isolé par firewall des autres réseaux : - Production CDE / Production hors CDE / Staging / Dev → 4 VLAN distincts - Aucun flux direct DB CDE ↔ DB hors CDE - Bastions pour l'accès opérateurs, avec session recording

    Key management · HSM et rotation

    HSM (Hardware Security Module)

    Stockage matériel des clés cryptographiques. La clé ne sort jamais du HSM. Le code applicatif demande "chiffre/déchiffre ceci avec la clé K42" mais ne voit jamais K42.

    Standards : FIPS 140-2 niveau 3 ou supérieur. Solutions : Thales Luna, AWS CloudHSM, Azure Dedicated HSM.

    Rotation des clés

    Imposée par PCI DSS Requirement 3.6 : - Clés de chiffrement de données : rotation au moins annuelle - Clés exposées (compromission soupçonnée) : rotation immédiate - Versioning des clés dans le ciphertext (v1, v2...) pour pouvoir déchiffrer l'historique

    PAN chiffré : {v: 3, iv: "...", data: "..."}
                 ↑ version de clé utilisée

    Patching et CVE · Requirement 6

    Toute CVE critique doit être patchée sous 30 jours. En pratique on vise plus court sur les composants exposés.

    Outils : - Dependabot / Renovate sur GitHub - OWASP Dependency-Check en CI - Snyk ou GitHub Advanced Security pour la SCA - Trivy pour scanner les images Docker

    Workflow type : 

    CVE détectée → ticket sécu → analyse impact → patch → tests → déploiement
                                  ↑
                               SLA : 30j critique, 90j high

    Cas concrets

    Que faire si on découvre un PAN dans les logs de prod ?

    Incident PCI DSS. Procédure : (1) identifier la fenêtre temporelle et le périmètre concerné, (2) purger immédiatement les logs concernés (en gardant trace de la purge), (3) corriger le code qui a loggué le PAN, (4) déclarer l'incident en interne (RSSI, DPO si données personnelles aussi), (5) postmortem avec actions correctives pérennes. Selon l'ampleur : notification du QSA / banques / clients.

    Faut-il chiffrer même les données dans les volumes Kafka ?

    Idéalement, ne pas mettre de PAN brut dans Kafka. Tokeniser avant. Si impossible : chiffrer au niveau message (clé applicative, déchiffrée juste au moment du besoin) + chiffrer les volumes (LUKS, disque géré). PCI DSS exige le chiffrement at rest.

    Heap dump auto sur OOM en prod, c'est compatible PCI DSS ?

    Oui mais avec précautions : (1) volume de destination chiffré (LUKS), (2) accès restreint root, (3) logué dans l'audit trail, (4) destruction après analyse documentée. Sans ces précautions : risque de fuite massive de PAN/CVV dans un fichier .hprof lisible.

    Le CVV est-il dans les messages Kafka ?

    JAMAIS. Le CVV ne survit pas à l'autorisation initiale. Il rentre dans la requête HTTPS du marchand, est envoyé à la banque, et est détruit en mémoire immédiatement après. Aucune persistance, aucun log, aucun message Kafka.

    Mon Tomcat log les requêtes en POST. Y a-t-il un risque ?

    Si le body POST contient un PAN ou un CVV en clair → oui. Les access logs Tomcat ne loggent normalement que le path et les paramètres URL (pas le body), donc les POST sont OK par défaut. Mais si quelqu'un a activé request.dumper ou un filter de debug qui log le body, danger. Toujours vérifier la config.

    Comment réduire le périmètre CDE de mon SI ?

    Tokenizer dès la réception. Le composant qui reçoit le PAN du marchand le transforme immédiatement en token via un vault. Le reste du SI ne voit que des tokens et sort du CDE. C'est la stratégie qui réduit le plus le coût audit / certification.

    Points clés

    • PCI DSS impose 12 requirements, dont les plus impactants côté tech : protection des données stockées, chiffrement transit, RBAC, audit log immuable, patching
    • Tokenization est la stratégie qui réduit le CDE et donc le coût de mise en conformité
    • CVV, PIN, full track : jamais persistés, jamais loggués, jamais dans Kafka
    • PAN masqué (6+4) suffit pour les logs et metrics
    • Aucune donnée carte dans une URL (access logs)
    • TLS 1.2+ obligatoire, viser A+ SSL Labs
    • HSM ou KMS pour les clés, rotation annuelle minimum
    • Heap dumps sur volume chiffré, accès tracé
    • Audit log dédié pour tout accès au PAN, conservation 1 an
    • CVE critique patchée sous 30 jours
